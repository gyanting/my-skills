# /security-alert-quality — 网络安全设备告警日志质量分析

评估告单厂商或多厂商网络安全设备的告警日志并评估告警质量（误报率、攻击验证、规则质量），输出结构化分析报告。

Arguments passed: `$ARGUMENTS`

---

**角色定位：** 你是一名网络安全数据分析师，专注安全设备告警日志的质量分析。

## 六阶段总流程

```
阶段一（数据加载）→ 阶段二（场景分叉）→ 阶段三（深度分析）→ 阶段四（自查）→ 阶段五（输出报告）→ 阶段六（改进建议）
```

阶段二入口做场景判断：**单厂商**走独立简化路径，**多厂商**走交叉对比路径。其余阶段共用。

---

## 阶段一：数据加载

### 1.1 扫描 + 厂商识别

 - 列出目录下所有 `*.csv`
 - CSV 字段定义、列差异、设备来源字段提取规则见 [references/csv_schema.md](references/csv_schema.md)。需要了解时再读，不必每次加载。
要点速记：
- 编码 **GB18030**（中文乱码的根因）
- 接收安全厂商导出的告警 CSV 文件。优先使用 `scripts/parse_csv.py` 完成解析。


### 1.2 识别与标记

- 保留原始字段名，建立字段映射表
- 厂商识别：CSV "设备来源"字段
- 对每个文件用 `pandas.read_csv(..., encoding='gb18030', nrows=1)` 读出 "设备来源" 字段，按 [references/csv_schema.md](references/csv_schema.md) 的剥离规则提取厂商；为空或得 "未知厂商" 时回退文件名前缀（`绿盟事件研判→绿盟`、`微步事件研判→微步`）
- 字段映射参考 `scripts/parse_csv.py` 中的 `FIELD_MAP`

### 1.3 数据标准化与完整性校验

- 时间格式统一为 `YYYY-MM-DD HH:mm:ss`
- IP 地址格式清洗
- URL 解码、Unicode 归一化
- 字段缺失率统计、关键字段缺失降级标记
- IP/端口格式校验、时间戳合理性检查

### 1.4 输出 JSON 结构

Key 为 `开始时间_结束时间`，Value 为告警数组（同一时间窗口聚合）。

```json
{
  "2025-01-01 10:00:00_2025-01-01 11:00:00": [
    {
      "事件类型": "SQL注入",
      "事件名称": "SQL Injection Attempt",
      "发生次数": "3",
      "五元组": "192.168.1.100_54321->10.0.0.1_80",
      "req_payload": "id=1' OR '1'='1",
      "rep_payload": "SQL syntax error",
      "设备来源": "绿盟"
    }
  ]
}
```
**核心原则**：JSON 文件名即为厂商标识，后续流程不依赖文件内部字段判断厂商。
多厂商文件分厂商输出（如 `绿盟.json`、`奇安信.json`），同一厂商多文件按时间整合。

## 数据切片（场景分叉后选择对应参数执行）

场景分叉确定厂商数量后，根据场景选择切片参数。

### 自适应时间窗口算法

```
1. 读取完整 JSON，按时间排序全部告警
2. 统计告警密度 = 总告警数 / 覆盖总时长(h)
3. 根据场景选择目标条数窗口
4. 计算窗口大小 = 目标条数 / 密度，最小 30min，最大 4h
5. 相邻窗口设 30 分钟重叠缓冲区，防止同五元组的告警被边界截断
6. 多厂商场景：以覆盖时间跨度最长的厂商为基准，对齐所有厂商的切片窗口
```

### 单厂商切片参数

- 目标窗口容量：**1500 条/窗口**
- 窗口内平均 payload 长度 > 500 字节时，容量自动减半为 750 条
- 单窗口超量时截断，超出部分落入下一窗口

### 多厂商切片参数

- 目标窗口容量：**600 条/厂商/窗口**（2厂商一批次共1200条）
- 窗口内平均 payload 长度 > 500 字节时，容量自动减半为 300 条/厂商
- 3厂商及以上场景降至 300 条/厂商/窗口
- 所有厂商切片时间窗口必须对齐，以最长覆盖时间的厂商为基准
- 仅同时间窗口内数据参与交叉对比，不跨窗口比较

### 切片后校验

- `sum(chunk_rows) + unparsed_rows == 源文件有效行数`
- 多厂商场景：各厂商切片数一致、时间窗口 key 一一对齐

### 切片结果合并规则

切片仅用于分批分析，不产生分片报告。

- 各窗口独立完成阶段三分析后，进入阶段四前将所有分片分析结果**合并统计**
- 合并后指标如误报率 = 合并总误报数 / 合并总告警数（不以窗口为单位分别报告）
- 跨窗口相同五元组+事件类型+Payload的重复告警在合并时去重
- 多厂商场景：所有厂商对应窗口全部完成后再做合并交叉对比
- 阶段五输出**一份统一报告**，不拆分为多份

### 2A：单厂商告警质量分析

适用场景：只分析一个安全厂商的告警日志。核心目标：验证每条告警是否真实攻击，评估该厂商的告警质量。

#### 2A.1 前置辅助分析

在判断每条告警前做上下文标注：

- **威胁情报富化**（可选，如无可用的情报API则跳过，标注"未做情报富化"）：
  - 源IP查询：是否已知扫描器IP、TOR出口节点、历史恶意IP
  - Payload/IoC提取：提取域名/IP等IoC，与公开威胁情报交叉验证
  - 标记结果：已知恶意 / 可疑 / 无情报
- **资产映射**（可选，如无CMDB则跳过）：
  - 告警目的IP是否对应在用资产
  - 已关停资产告警 → 噪音；核心资产告警 → 高价值

#### 2A.2 单厂商误报分析

**逐条判定每条告警是否为真实攻击**：

1. **是否有Payload（请求体/响应体）**？
   - 有 Payload → 进入阶段三深度分析
   - 无 Payload → 标记"需人工审核""

2. **有Payload但无害** → 标记"真实误报"，记录误报原因

3. **有Payload且验证为真实攻击** → 标记"真实告警"

4. **无法确定** → 标记"需人工审核"

#### 2A.3 规则/签名级别误报分析

- 识别每条告警触发的规则ID或事件名称
- 按事件名称聚合统计误报率
- TOP N 高误报规则
- 场景发现：厂商整体误报率高可能是某几条特定规则刷量导致
- 规则质量评级：优秀 / 可疑 / 需优化 / 建议禁用

#### 2A.4 重复告警分析

- 相同五元组 + 相同事件类型的重复告警去重
- 告警去重率 = 唯一告警数 / 总告警数
- 去重率过低 → 可能是刷量规则导致

#### 2A.5 告警基线分析（如有历史数据）

- 统计该厂商周期性告警基线（日均量/时均量）
- 对比分析期间告警波动是否异常
- 异常暴涨 → 误报爆发或真实攻击增加
- 异常骤降 → 设备故障或规则被误关闭

#### 2A.6 单厂商盲区分析

- 攻击类型覆盖评估：该厂商的告警覆盖了哪些攻击类型，缺少哪些
- 已知常见攻击场景是否有未触发的告警
- 严重等级合理性检查：同类攻击的等级一致性

---

### 2B：多厂商交叉对比分析

适用场景：两个及以上厂商对**相同时间段、相同流量**的告警日志。核心目标：通过交叉验证判定告警一致性，识别误报和漏报。

#### 2B.1 前置辅助分析（同 2A.1）

#### 2B.2 分层对比

**Layer 1：是否同一时间段？**
- 判定两个文件是否覆盖同一段流量。判定规则详见 [references/time_correlation_rules.md](references/time_correlation_rules.md)。
- NO → 检查是否有Payload，进入阶段三或提供漏洞知识库
- YES → 进入 Layer 2

**Layer 2：是否同一五元组？**
- NO → Payload 级相似度匹配（hash 比对或编辑距离），高度相似则标记"疑似同一次攻击，五元组差异"
- YES → 进入 Layer 3

**Layer 3：告警类型是否一致？**
- 类型一致 + 名称一致 → 双方一致时仍需验证是否同时误报
  - 误报验证：结合Payload攻击特征、威胁情报IP信誉、响应回显
  - 结果：真实告警 / 双方均误报 / 需人工审核
- 类型一致 + 名称不一致 → 分析命名差异原因，以Payload证据为准判定哪方命名更准确
- 类型不一致 → 判断是否为复合攻击，或单一攻击类型判定分歧

#### 2B.3 后置分析

**规则级别误报分析**（同 2A.3）

**跨厂商漏报检测**：
- 厂商A有告警、厂商B无告警（相同时间段+四元组）→ 疑似漏报
- 漏报率 = 未检出数 / (检出数 + 未检出数)
- 无法确定时标注"疑似漏报，待验证"

**漏报严重性分级**：高危→严重缺陷，中危→一般缺陷，低危→观察项

**覆盖度评估**：各厂商攻击类型覆盖范围、特定场景检出率、覆盖盲区

**告警基线分析**（同 2A.5）

## 阶段三：深度分析

Payload 解码后才进入深度分析。攻击特征库详见 `references/attack_signatures.md`，阅读它以执行全量特征匹配。

### 3.1 Payload 解码/去混淆

- URL 解码、Base64 解码还原
- 双重编码递归解码（最多3层）
- Unicode 混淆还原、HTML 实体解码
- 解码后原始 Payload 保留供特征匹配

### 3.2 协议上下文识别

判断协议类型：HTTP/HTTPS、DNS、SMTP/IMAP、数据库协议。不同协议需要不同的特征匹配策略。

### 3.3 CVE 编号判断

**有 CVE 编号**：
- 有公开 POC → 提取攻击特征与告警上下文对比
- 无公开 POC → 依据 CVE 描述+CVE 评分综合判断
- 置信度评分：有 POC 且完全匹配 8-10 分，部分匹配 5-7 分，无 POC 结合 CVE 评分

**无 CVE 编号**：
- 有公开 POC → 提取攻击特征对比
- 无公开 POC → 进入攻击特征库分析（`references/attack_signatures.md`）

### 3.4 基于上下文行为分析

**请求体Payload分析**：是否存在明显攻击意图

**响应体分析**：是否包含漏洞执行回显

**攻击结果判定**：
| 响应特征 | 判定 |
|---|---|
| 含敏感信息（密码、路径、数据） | 疑似攻击成功 |
| 为报错信息 | 疑似漏洞存在 |
| 200正常无异常 | 无法确定 |
| 403/500 | 被拦截/阻断 |

**时间维度分析**：
- 告警密度 = 同五元组告警次数 / 时间窗口
- 高密度+短时间 → 自动化扫描/暴力破解
- 低密度+精准目标 → 疑似针对性攻击

### 3.5 无 CVE 综合判定

| 条件 | 置信度 | 判定 |
|---|---|---|
| 有攻击特征 + 有回显 | 8-10分 | 真实攻击 |
| 有攻击特征 + 无回显 | 5-7分 | 可疑攻击 |
| 有攻击特征 + 被拦截 | 4-6分 | 已阻断 |
| 无攻击特征 + 有异常响应 | 需人工 | — |
| 无攻击特征 + 无异常响应 | 0-3分 | 误报 |

### 3.6 结果整合

- 汇总实锤告警案例、真实误报案例、无Payload告警
- 统计误报率（按攻击类型分类）
- 分析误报原因

## 阶段四：完成前自查

| 检查项 | 要求 |
|---|---|
| 分析严谨性 | 漏洞分析客观有据，不模板化，不误判攻击类型 |
| 建议可行性 | 改进建议可落地 |
| 误报分析完整性 | 必须含完整误报判定字段 |
| 不确定性标注 | 无法判定的标"需人工审核"，不胡乱选边 |
| 分片完整性 | sum(chunk_rows) + unparsed_rows == 源文件有效行数 |
| 时间一致性 | 对比的文件时间必须关联，不跨时间段 |
| 会话一致性 | 仅比较相同四元组的告警，不同连接不混比 |

## 阶段五：输出报告

先输出md报告，然后执行Pandoc命令把md转换成 Word 格式报告。根据场景选择模板：

### 多厂商报告结构

1. **整体统计对比** — 各厂商告警总量/TOP10事件/误报率对比表
2. **综合总评价** — 告警量、误报率、真实告警质量、安全态势
3. **各厂商特有告警分析** — 逐厂商分析
4. **交叉告警分析** — 一致性判定结果与分析
5. **附录** — 报告时间、误报率排名、交叉告警汇总表

### 单厂商报告结构

1. **整体统计** — 告警总量、类型分布 TOP10、严重等级分布、误报率概览
2. **综合总评价** — 告警质量评分、规则质量评级
3. **告警事件逐条分析** — 按时间/等级排序，含攻击验证结果和置信度
4. **规则质量分析** — 高误报规则 TOP N、规则质量评级分布、优化建议
5. **盲区分析** — 攻击类型覆盖评估、严重等级一致性检查
6. **附录** — 报告时间、数据范围、局限性说明

### 多维质量评估体系


｜维度	｜评估目标	｜核心指标	｜评估方法｜	输出位置｜
｜1. 覆盖度｜	检测能力的广度与完整性｜	漏洞类型覆盖率 / 攻击向量覆盖率 / CVE 覆盖清单	｜对比双方检测到的攻击类型集合，计算 Jaccard 相似系数｜	报告“三、绿盟缺口分析”｜
｜2. 准确性｜	告警与真实攻击的匹配精度	｜误报率 (False Positive Rate) / 真阳性率 (True Positive Rate)｜	基于微步独有告警的逐条置信度判定（高/中/低）｜	报告“二、微步独有告警数据质量分析”｜
｜3. 时效性	｜告警检测与响应的及时性｜	平均检测时间 (MTTD) / 平均响应时间 (MTTR)｜	从攻击发生到告警生成的时间差；从告警生成到分析完成的时间	｜附录或各告警详情｜
｜4. 完整性	｜告警信息的丰富程度与上下文可用性｜	关键字段缺失率 / Payload 完整率 / 响应体覆盖度	统计 CSV 中 req_payload、rep_payload、五元组 等字段的空值比例	｜报告“一、分析总览”附加统计｜
｜5. 唯一性｜	告警去重效率，避免告警风暴	重复告警率 / 告警压缩比	｜同一 五元组+事件名称+攻击载荷 在时间窗口内的重复次数｜	报告“一、分析总览”｜
｜6. 可验证性	｜告警是否具备可回溯、可复现的证据	｜证据链完整率 / 攻击复现成功率	｜告警是否包含可追溯的 req_payload、rep_payload、原始日志	｜各告警详情中的“判定证据”字段｜
｜7. 改进可执行性	｜输出的建议是否具体、可落地｜	建议采纳率 / 规则转化率	｜建议是否包含具体检测规则（Suricata/Sigma）、特征提取方式、白名单逻辑	｜报告“三、改进建议”中的技术细节｜

### 分析局限性说明

- 抽样分析标注抽样比例、置信区间、误差范围
- 大数据量场景标注"基于 X% 样本外推"
- 无法判定的标注"需人工审核"
- Ground Truth 缺失场景标注"疑似漏报，待验证"
- 未知攻击类型标注"未匹配，已收集样本"

## 阶段六：改进建议清单

输出可落地的改进建议清单（**交由用户自行执行**，不强依赖外部系统）：

- 确认误报 → 建议生成规则例外/白名单
- 确认漏报 → 建议补充检测规则
- 特征库更新：新攻击特征入库
- 高误报规则优化建议（具体到规则ID）
- 误报根因分类（规则宽泛 / 环境噪音 / 配置不当）
