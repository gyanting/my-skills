## 快速判定摘要

以下为脚本/LLM自动判定使用的结构化规则，详细分析见后续章节。

### 真阳性指标

| 置信度 | 特征 | 说明 |
|--------|------|------|
| 高 | 请求含 `;cat `, `;ls `, `;id`, `;whoami`, `;uname` | Shell命令注入 |
| 高 | 请求含 `\|cat `, `\|ls `, `` `id` ``, `$(whoami)` | 管道/子shell注入 |
| 高 | 请求含 `eval(`, `exec(`, `system(`, `passthru(` | PHP代码执行函数 |
| 高 | 请求含 `Runtime.getRuntime`, `ProcessBuilder` | Java命令执行 |
| 高 | 请求含 `__proto__`, `constructor`, `thenable` + React Flight 格式 | CVE-2025-55182 React RCE |
| 高 | 请求含 SOAPAction + AddPortMapping + `/soap.cgi` | D-Link UPnP SOAP RCE |
| 中 | 请求含 `cmd.exe`, `/bin/bash`, `/bin/sh` | Shell调用尝试 |
| 中 | 响应体含 `root:x:`, `bin/bash`, 系统文件内容 | 注入成功回显 |

### 误报指标

| 置信度 | 特征 | 说明 |
|--------|------|------|
| 高 | 请求体中 Shell 元字符出现在 JSON/XML 数据值中 | 结构化数据中的字面量 |
| 高 | curl/wget 后跟正常URL（非内网地址） | 正常HTTP客户端行为 |
| 中 | `eval(` 出现在合法JS库代码中 | 前端正常脚本 |
| 中 | 参数经严格白名单/类型校验后透传 | 业务逻辑安全 |
| 低 | 编码后payload解码确认无攻击意图 | 编码本身不代表攻击 |

### 判定优先级

1. 先查 CVE 特征（React Flight / SOAP AddPortMapping / 已知漏洞签名）
2. 再查 Shell 元字符 + 上下文（是JSON字面量还是实际注入点）
3. 最后查响应体回显 → 有回显=高置信度真阳性，无回显降级

## 技能概述

网络安全设备通过内置规则检测命令注入攻击，但由于规则配置、检测逻辑、业务场景等因素，容易产生大量误报。本cookbook提供系统化的命令注入误报识别方法，帮助分析人员准确判定告警真实性。

## 角色定位

你是一名专业的安全告警研判分析师，专注于命令注入类告警的误报识别与判定。

## 分析输入

### 数据来源

从网络安全设备告警日志CSV中提取以下关键字段：

| 字段 | 用途 | 说明 |
|------|------|------|
| 请求体 | 分析注入Payload | 包含用户输入的参数内容 |
| 响应体 | 判断注入结果 | 包含服务器返回内容，用于验证注入是否成功 |
| 响应码 | 判断请求状态 | 200/404/500等状态码分析 |
| 载荷 | 分析流量特征 | 网络流量中的原始数据 |

### 编码绕过识别

在分析请求体时，必须检测以下编码方式（按检测优先级排序）：

| 优先级 | 编码类型 | 检测规则 | 解码方法 | 示例 |
|--------|----------|----------|----------|------|
| 1 | URL编码 | `%[0-9A-Fa-f]{2}` | urldecode | `%3B%20ls` → `; ls` |
| 2 | Base64 | `/^[A-Za-z0-9+/]+=*$/` 且len%4==0且len>=4 | base64_decode | `OyBscw==` → `; ls` |
| 3 | Hex编码 | `\\x[0-9A-Fa-f]{2}` | hex2bin | `\x3b\x20\x6c\x73` → `; ls` |
| 4 | Unicode编码 | `\\u[0-9A-Fa-f]{4}` | unicode_decode | `\u003b\u0020\u006c\u0073` → `; ls` |
| 5 | HTML实体编码 | `&#\d+;|&#x[0-9A-Fa-f]+;` | html_entity_decode | `&#59;&#32;&#108;&#115;` → `; ls` |
| 6 | UTF-7编码 | `/^\+[A-Za-z0-9+-]+-?\s*$/` | UTF-7解码 | `+AGYAbgAgAGwAdwA= -` → `fn lw` |
| 7 | Punycode编码 | `^xn--[A-Za-z0-9-]+$` | Punycode解码 | `xn--ls-zhx5a` |
| 8 | Base32 | `/[A-Z2-7]+=*$/` 且len%8==0 | base32_decode | `MFRGGZDFMY====` |
| 9 | 混合编码 | 多种编码特征组合 | 逐层解码 | `URL+Base64` |
| 10 | 双重/多重编码 | 多层编码特征 | 多次解码 | `%253B` → `%3B` → `;` |

**检测流程：**
1. 从请求体提取参数值
2. 按优先级依次检测编码特征
3. 如果匹配，立即进行解码
4. 对解码后的内容再次检测（处理多重编码）
5. 循环步骤2-4，最多10层或直到连续3次无编码匹配

### 场景1：业务参数含Shell元字符

**特征识别：**
- 参数值包含`;`、`|`、`&`、`||`、`&&`、`` ` ``、`$()`等Shell元字符
- 后端无命令执行函数调用

**误报判定依据：**
1. 参数分析：参数值包含Shell元字符
2. 函数调用分析：后端未调用`system()`、`exec()`、`popen()`等命令执行函数
3. 上下文验证：参数仅用于日志记录、文件路径拼接、业务文本展示等非命令执行场景

**典型Payload示例：**
```
参数名: filename
参数值: report;error;2024.log
参数名: message
参数值: Note: A | B | C
```

**误报原因：**
扫描器或WAF的正则规则过于宽泛，只要检测到Shell命令分隔符就触发告警。但实际上，后端并没有将这些参数拼接到命令执行函数中，这些符号仅仅被当作普通字符串处理。

**判定逻辑（置信度：中）：**
```
IF 参数值包含Shell元字符（;、|、&、`、$()等）
AND 响应正常，无命令执行迹象
AND 推断后端无命令执行函数调用（无法直接确认，通过响应推断）
AND 参数仅用于非命令执行上下文（日志、路径、文本展示等）
THEN 误报 = True
```

### 场景2：安全原生API替代系统命令

**特征识别：**
- 使用安全原生API而非系统Shell命令
- API包括`os.rename()`、`Files.copy()`、`fs`模块等

**误报判定依据：**
1. API调用识别：使用安全原生API
2. 执行方式：不通过Shell执行，直接调用系统API
3. 安全性：原生API通常不支持命令注入

**安全原生API特征：**

| 语言 | 安全API | 危险API |
|------|---------|---------|
| Python | `os.rename()`, `shutil.copy()`, `pathlib` | `os.system()`, `subprocess.call(shell=True)` |
| Java | `Files.copy()`, `Files.move()` | `Runtime.exec()`, `ProcessBuilder` |
| Node.js | `fs.rename()`, `fs.copyFile()` | `child_process.exec()` |
| PHP | `rename()`, `copy()` | `exec()`, `shell_exec()`, `system()` |
| Go | `os.Rename()`, `io.Copy()` | `exec.Command(shell)` |

**典型示例：**
```
Python: os.rename(old_path, new_path)
Java: Files.copy(source, target)
Node.js: fs.renameSync(oldPath, newPath)
```

**误报原因：**
静态代码审计工具（SAST）在进行污点追踪时，发现"用户输入 → 敏感操作"，但没有深入理解底层调用的具体实现。只要不是直接调用系统Shell，这类原生API通常是安全的，不会引发命令注入。

**判定逻辑（置信度：高）：**
```
IF 操作使用安全原生API（os.rename、Files.copy、fs模块等）
AND API不通过Shell执行
AND 用户输入仅作为参数传递
THEN 误报 = True
```

### 场景3：参数化执行且shell=False

**特征识别：**
- 命令以数组传递
- `shell=False`参数
- 用户输入作为参数而非命令

**误报判定依据：**
1. 调用方式：命令以数组/列表形式传递
2. shell参数：`shell=False`或等效配置
3. 输入位置：用户输入作为参数而非命令部分

**参数化执行特征：**

| 语言 | 安全调用示例 | 危险调用示例 |
|------|-------------|-------------|
| Python | `subprocess.call([cmd, arg1, arg2], shell=False)` | `subprocess.call(cmd + arg, shell=True)` |
| Java | `ProcessBuilder(cmd, arg1, arg2)` | `Runtime.exec("cmd " + arg)` |
| Node.js | `child_process.spawn(cmd, [arg1, arg2])` | `child_process.exec(cmd + " " + arg)` |

**典型示例：**
```
# Python安全调用
subprocess.run(["grep", "pattern", filename], shell=False)

# 用户输入: filename = "; rm -rf /"
# 实际执行: grep "pattern" "; rm -rf /"
# 结果: 尝试查找名为"; rm -rf /"的文件，不会执行删除命令
```

**误报原因：**
这种写法下，即使用户输入了`; rm -rf /`，系统也只会把它当成一个普通文件名去查找，而不会将其解析为多条命令。扫描器往往无法识别`shell=False`这一关键的安全配置，从而产生误报。

**判定逻辑（置信度：高）：**
```
IF 命令以数组形式传递
AND shell参数设置为False或等效配置
AND 用户输入作为参数而非命令部分
AND 参数不会触发命令分隔符解析
THEN 误报 = True
```

### 场景4：严格白名单/类型强转

**特征识别：**
- 参数经`int()`强转或枚举校验
- 响应含"参数错误"等提示

**误报判定依据：**
1. 参数验证：参数经过类型强转或白名单校验
2. 响应特征：返回参数验证错误
3. Payload影响：Payload未能通过验证

**参数验证方式：**

| 验证方式 | 特征 | 示例 |
|----------|------|------|
| 类型强转 | int()、float()等 | `int(user_input)` |
| 白名单校验 | 枚举值检查 | `if value not in [1, 2, 3]` |
| 正则验证 | 格式验证 | `re.match(r'^\d+$', value)` |
| 范围检查 | 数值范围 | `if value < 0 or value > 100` |

**典型示例：**
```
请求: GET /api/process?id=1;whoami
响应: {"error": "参数必须是正整数"}

代码: user_id = int(request.args.get('id'))
```

**误报原因：**
扫描器发送了包含Payload的请求并收到了HTTP 200响应（可能是业务层返回的"参数不合法"提示）。由于缺乏对响应体内容的深度语义分析，扫描器误以为Payload成功逃逸并执行了。

**判定逻辑（置信度：高）：**
```
IF 参数经过类型强转或白名单校验
AND 响应返回参数验证错误
AND Payload未能通过验证（如非整数、不在白名单内）
AND 无命令执行迹象
THEN 误报 = True
```

### 场景5：防御性转义被忽略

**特征识别：**
- 代码中调用了转义函数
- 包括`escapeshellarg`、`escapeshellcmd`、`shlex.quote`、`pipes.quote`等
- 或有自定义过滤逻辑

**误报判定依据：**
1. 转义函数调用：识别安全转义函数
2. 转义效果：特殊字符被正确转义
3. 执行安全：转义后参数安全传递

**转义函数特征：**

| 语言 | 转义函数 | 效果 |
|------|----------|------|
| PHP | `escapeshellarg()` | 单引号包裹并转义单引号 |
| PHP | `escapeshellcmd()` | 转义shell元字符 |
| Python | `shlex.quote()` | Shell安全引用 |
| Python | `pipes.quote()` | Shell安全引用 |
| Ruby | `Shellwords.escape()` | Shell安全引用 |
| Node.js | 需自定义 | 正则替换 |

**典型示例：**
```
Python:
import shlex
safe_arg = shlex.quote(user_input)  # "; rm -rf /" → "'; rm -rf /'"
subprocess.call(["ls", safe_arg], shell=False)
```

**误报原因：**
扫描器的污点分析通常只追踪数据流，不评估数据是否经过了安全变换。经过`escapeshellarg()`包装后的输入，即使包含`; rm -rf /`，也会被转义为安全的字符串参数。扫描器未能识别这类安全函数的效果。

**判定逻辑（置信度：高）：**
```
IF 代码调用安全转义函数（escapeshellarg、shlex.quote等）
AND 特殊字符被正确转义
AND 转义后参数安全传递
THEN 误报 = True
```

**注意：** 从日志中无法直接确认是否调用了转义函数，需通过响应特征和上下文推断。

### 场景6：日志记录误判

**特征识别：**
- 告警关联的函数是日志类
- 包括`log`、`logger`、`print`、`console.log`等
- 而非命令执行类

**误报判定依据：**
1. 函数识别：关联函数为日志函数
2. 数据流分析：用户输入流向日志系统而非命令执行函数
3. 无执行风险：仅记录数据，不执行命令

**日志函数特征：**

| 函数类型 | 语言 | 示例 |
|----------|------|------|
| 日志记录 | Python | `logging.info()`, `print()` |
| 日志记录 | Java | `log.info()`, `System.out.println()` |
| 日志记录 | JavaScript | `console.log()` |
| 日志记录 | PHP | `error_log()` |

**命令执行函数对比：**

| 命令执行函数 | 语言 | 危险级别 |
|-------------|------|---------|
| system() | PHP/C/Python | 高 |
| exec() | PHP/Java | 高 |
| shell_exec() | PHP | 高 |
| popen() | C/PHP | 高 |
| os.system() | Python | 高 |
| subprocess.call(shell=True) | Python | 高 |

**误报原因：**
静态分析工具将日志函数的参数传递误判为命令执行函数的参数传递。虽然用户输入确实出现在了代码中，但它流向的是日志系统而非`system()`函数。扫描器的污点分析精度不足，无法区分不同的汇聚点。

**判定逻辑（置信度：中）：**
```
IF 告警关联的函数为日志类（log、logger、print、console.log等）
AND 非命令执行类（system、exec、popen等）
AND 用户输入仅用于日志记录
AND 无后续命令执行操作
THEN 误报 = True
```

### 场景7：执行结果不可控/无回显

**特征识别：**
- 命令输出被忽略（`stdout=None`, `stderr=None`）
- 返回值未被处理
- 无带外通道（DNS外带、HTTP回连）

**误报判定依据：**
1. 输出处理：命令输出被忽略或未返回给用户
2. 返回值处理：命令返回值未被检查或利用
3. OOB通道：无带外数据外泄通道

**无回显特征：**

| 模式 | 示例 | 说明 |
|------|------|------|
| 忽略输出 | `popen(cmd, stdout=None)` | 输出被丢弃 |
| 后台执行 | `cmd &` | 后台运行，输出丢失 |
| 异步执行 | `exec_async(cmd)` | 异步执行，无立即返回 |
| 无处理返回值 | `system(cmd)` | 返回值未检查 |

**典型示例：**
```
Python:
subprocess.run(["command", user_input], stdout=subprocess.DEVNULL, stderr=subprocess.DEVNULL)
# 命令执行但输出被忽略，攻击者无法获取结果
```

**误报原因：**
扫描器仅检测到"用户输入→命令执行"的污点链路即报告高危漏洞，未评估实际可利用性。即使攻击者能注入额外命令，由于命令输出不返回且无带外通道（如DNS外带），无法窃取任何敏感信息。

**判定逻辑（置信度：中）：**
```
IF 命令执行存在用户可控输入
AND 命令输出被忽略（stdout=None、stderr=None或等效配置）
AND 返回值未被处理或利用
AND 无带外数据外泄通道（DNS外带、HTTP回连等）
AND 响应无敏感信息泄露
THEN 误报 = True
```

### 场景16：延迟/OOB受环境干扰

**特征识别：**
- 网络波动被误判为延时注入
- DNS外带请求被网络设备拦截

**误报判定依据：**
1. 延迟分析：响应时间差异来自网络波动
2. OOB分析：DNS/HTTP外带被中间网络设备拦截
3. 环境因素：防火墙、DNS过滤等安全设备干扰

**环境干扰特征：**

| 干扰类型 | 特征 | 误报原因 |
|---------|------|---------|
| 网络波动 | 响应时间随机变化 | 误判为延时注入 |
| DNS过滤 | DNS请求被拦截 | OOB证据缺失 |
| 防火墙规则 | 出站连接被阻断 | OOB失败 |
| 负载均衡 | 请求分发到不同节点 | 响应差异 |

**典型示例：**
```
请求: POST /api/command
请求体: {"cmd": "echo hello; sleep 5"}
响应时间: 5200ms（网络延迟+处理时间）

对比请求: POST /api/command
请求体: {"cmd": "echo hello"}
响应时间: 4800ms（网络延迟+处理时间）
差异: 400ms（属于正常波动范围，非真实延时）
```

**误报原因：**
网络波动导致的自然延迟会被误判为"延时注入成功"；而DNS请求被中间网络设备拦截，会导致扫描器收不到回调，从而漏报或在某些逻辑下产生错误的推断。

**判定逻辑（置信度：低）：**
```
IF 仅凭响应时间差异判定（无明显延时特征）
AND 延迟时间在正常波动范围内
AND 无其他OOB证据（DNS外带、HTTP回连等）
AND 可能有环境干扰因素（防火墙、DNS过滤等）
THEN 误报 = True
```

```
响应: {"success": true, "queue_id": "xxx"}

后台处理:
worker.execute_queue_commands()  // 后台工作进程从队列取出并执行

说明: 命令被添加到队列，后续批量执行时才可能触发检测
```

**误报原因：**
在批量操作或队列场景中，用户的单个请求不会立即触发命令执行，而是被收集到队列中等待后续处理。此时触发的告警可能是由于批量处理时多个命令合并执行导致的，单个Payload无法单独利用。

**判定逻辑（置信度：中）：**
```
IF 命令在后台批量或队列执行
AND 用户输入被收集而非立即执行
AND 检测在批量处理时触发
THEN 误报 = True（单个请求无法利用）
```

## 非误报判定标准

### 真实命令注入告警特征

若不满足上述任何误报场景，且存在以下明确注入证据，则判定为**真实告警**：

#### 1. 命令执行成功且有回显

**特征：**
- Payload成功执行
- 响应体包含命令执行结果

**判定：**
```
IF Payload包含命令注入特征（;、|、&等）
AND 响应体包含命令执行结果（如文件列表、系统信息等）
AND 结果超出正常业务范围
THEN 真实告警 = True
```

#### 2. 延时注入可验证

**特征：**
- 包含延时命令的Payload产生可测量的延迟
- 延迟时间可控且明显异常

**判定：**
```
IF Payload包含延时命令（sleep、ping等）
AND 响应时间明显延迟（与命令参数正相关）
AND 延迟时间远超正常响应范围
THEN 真实告警 = True
```

#### 3. 带外通道回连成功

**特征：**
- 建立带外连接（DNS、HTTP）
- 接收到目标服务器响应

**判定：**
```
IF 建立带外连接（DNS解析、HTTP回调等）
AND 接收到目标服务器响应
AND 响应包含敏感信息或确认连接
THEN 真实告警 = True
```

#### 4. 文件操作成功

**特征：**
- 成功创建、修改或删除文件
- 文件操作结果可见

**判定：**
```
IF Payload包含文件操作命令
AND 文件操作成功执行（通过后续请求验证）
AND 文件内容或状态被改变
THEN 真实告警 = True
```

#### 5. 反向Shell建立

**特征：**
- 成功建立反向Shell连接
- 可执行后续命令

**判定：**
```
IF Payload包含反向Shell命令（bash -i、nc等）
AND 后续网络连接建立成功
AND 可通过连接执行命令
THEN 真实告警 = True
```

## 判定示例

### 示例1：误报 - 安全原生API替代系统命令

```
请求: POST /api/file/rename
请求体: {"old": "file1.txt", "new": "file2;ls.txt"}
响应: {"success": true}
分析: 使用Files.copy()安全API，非系统命令执行
判定: 误报（场景2）
```

### 示例2：误报 - 严格白名单/类型强转

```
请求: GET /api/process?id=1;whoami
响应: {"error": "参数必须是正整数"}
分析: 参数经int()强转验证，返回参数错误
判定: 误报（场景4）
```

### 示例3：误报 - 业务参数含Shell元字符

```
请求: POST /api/report/save
请求体: {"filename": "report;error;2024.log"}
响应: {"success": true}
分析: 文件名包含分号，但仅用于文件操作，无命令执行
判定: 误报（场景1）
```

### 示例4：误报 - 日志记录误判

```
请求: POST /api/log
请求体: {"message": "test;echo hello"}
响应: {"success": true}
分析: 参数仅用于日志记录，非命令执行函数
判定: 误报（场景6）
```

### 示例5：真实告警 - 命令执行成功且有回显

```
请求: POST /api/command
请求体: {"cmd": "cat /etc/passwd"}
响应: {
  "result": "root:x:0:0:root:/root:/bin/bash\ndaemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin..."
}
分析: 成功执行命令并返回敏感文件内容
判定: 真实告警（命令执行成功且有回显）
```

### 示例6：真实告警 - 延时注入可验证

```
请求1: POST /api/command
请求体: {"cmd": "ping -c 3 127.0.0.1"}
响应时间: 3000ms

请求2: POST /api/command
请求体: {"cmd": "echo test"}
响应时间: 50ms

分析: 延迟明显且可控（ping -c 3）
判定: 真实告警（延时注入可验证）
```

### 示例7：误报 - 参数绑定/DTO映射的自动类型转换

```
请求: GET /api/user?id=1;whoami
响应: {"id": 1, "name": "admin"}
分析: Spring MVC自动将id转为int类型，分号被忽略
判定: 误报（场景17）
```

### 示例8：误报 - 框架内置安全过滤器

```
请求: POST /api/command

响应: {"success": true}
分析: Laravel中间件过滤了HTML标签，命令未执行
判定: 误报（场景18）
```

### 示例9：误报 - 批量操作/队列命令场景

```
请求: POST /api/commands/add
请求体: {"command": "ls;whoami"}
响应: {"success": true, "queued": true}
分析: 命令被加入队列，单个请求无法立即执行
判定: 误报（场景19）
```

### 使用说明

1. **触发条件**：当用户需要分析命令注入类告警时使用此cookbook
2. **分析步骤**：
   - 提取请求体和响应体
   - 识别并解码编码内容
   - 首先检查高优先级场景（API明确、响应明确）
   - 按优先级检查误报场景
   - 如无匹配，检查真实命令注入特征
   - 输出判定结果和依据

3. **置信度评估**：
   - 高：明确匹配误报场景（API明确、响应明确）或真实命令注入特征（有回显、有OOB证据）
   - 中：部分匹配，需要额外上下文确认或推断类场景
   - 低：无明确特征，需要人工复核

## 注意事项

1. **编码识别优先级**：按URL编码 → Base64 → Hex → Unicode → HTML实体 → 多重编码的顺序检测
2. **编码检测流程**：最多解码10层或直到内容不再变化，防止解码循环
3. **上下文分析**：结合URL路径、参数名、业务场景综合判断
4. **响应分析**：重点分析响应体是否包含命令执行成功的证据
5. **误报场景覆盖**：确保覆盖所有19个误报场景
6. **场景优先级**：高优先级场景优先匹配，低优先级场景需要更多上下文验证
7. **无法判定处理**：对于无法判定的告警，标记为"待人工复核"并记录原因
8. **场景冲突处理**：多场景匹配时选择优先级最高的，或标记为"多场景匹配"
9. **环境因素**：注意容器环境、沙箱隔离、网络设备干扰等环境因素
10. **API调用识别**：区分安全原生API和危险命令执行函数
11. **参数化执行**：识别shell=False等安全配置
12. **防御机制**：识别转义函数、白名单校验等防御机制
13. **数据流分析**：区分日志记录、数据存储、命令执行等不同数据流
14. **OOB证据**：带外通道证据是判定真实命令注入的重要依据
15. **容器隔离**：容器环境中的命令注入风险需结合隔离级别评估
