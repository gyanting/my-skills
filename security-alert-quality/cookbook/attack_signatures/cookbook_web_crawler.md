## 快速判定摘要

以下为脚本/LLM自动判定使用的结构化规则，详细分析见后续章节。

### 真阳性指标（判定为非误报）

| 置信度 | 特征 | 说明 |
|--------|------|------|
| 高 | UA含 sqlmap/nikto/acunetix/nessus/burpsuite/appscan/awvs | 已知安全扫描工具 |
| 高 | URL含 /admin, /wp-admin, /phpmyadmin, /manager, /config | 敏感管理路径扫描 |
| 高 | URL含 .git/, .env, .svn/, WEB-INF/, .DS_Store | 敏感文件/目录探测 |
| 中 | 请求含 ../../etc/passwd 等路径遍历 | 伴随漏洞利用的扫描 |
| 中 | 同源IP短时间内请求 >50 个不同URI | 高频自动化扫描 |

### 误报指标（判定为误报）

| 置信度 | 特征 | 说明 |
|--------|------|------|
| 高 | UA含 googlebot/baiduspider/bingbot/sogou/yandex/duckduckgo | 搜索引擎爬虫 |
| 高 | UA含 facebookexternalhit/twitterbot/slackbot | 社交媒体预览爬虫 |
| 高 | UA含 Mozilla/5.0.*compatible.*crawler/bot/spider | 声明了爬虫身份 |
| 中 | 请求均为静态资源 (css/js/png/jpg/svg/ico) | SPA资源加载 |
| 中 | 请求间隔规则、遵循 robots.txt 模式 | 良性爬虫行为 |
| 中 | UA含 Microsoft-Delivery-Optimization | Windows Update P2P |
| 中 | 目的端口为 4070 + payload含 discover;013; | HID VertX设备发现 → 转IoT漏洞分析 |

### 判定优先级

1. 先查 UA → 搜索引擎/已知扫描器直接判定
2. 再查 URL 模式 → 敏感路径=真阳性，静态资源=误报
3. 最后查行为 → 高频+随机路径=扫描，低频+有规律=可能正常

## 技能概述

网络安全设备通过内置规则检测网络爬虫行为，但由于规则配置、检测逻辑、业务场景等因素，容易产生大量误报。本cookbook提供系统化的网络爬虫误报识别方法，帮助分析人员准确判定告警真实性。

## 角色定位

你是一名专业的安全告警研判分析师，专注于网络爬虫类告警的误报识别与判定。

## 分析输入

### 数据来源

从网络安全设备告警日志CSV中提取以下关键字段：

| 字段 | 用途 | 说明 |
|------|------|------|
| 请求体 | 分析请求特征 | 包含请求URL、参数内容 |
| 响应体 | 判断请求结果 | 包含服务器返回内容 |
| 响应码 | 判断请求状态 | 200/404/403/429等状态码分析 |
| 响应头 | 分析请求特征 | User-Agent、Referer、缓存标识等 |
| 载荷 | 分析流量特征 | 网络流量中的原始数据 |
| 源IP | 来源IP地址 | 用于IP白名单判断 |
| 目的IP | 目标IP地址 | 用于目标地址分析 |

### 编码绕过识别

在分析请求体时，必须检测以下编码方式（按检测优先级排序）：

| 优先级 | 编码类型 | 检测规则 | 解码方法 | 示例 |
|--------|----------|----------|----------|------|
| 1 | URL编码 | `%[0-9A-Fa-f]{2}` | urldecode | `%2e%2e%2f` → `../` |
| 2 | 双重URL编码 | `%25[0-9A-Fa-f]{2}` | 两次urldecode | `%252e%252e%252f` → `../` |
| 3 | Unicode编码 | `\\u[0-9A-Fa-f]{4}` | unicode_decode | `\u002e\u002e\u002f` → `../` |
| 4 | Hex编码 | `\\x[0-9A-Fa-f]{2}` | hex2bin | `\x2e\x2e\x2f` → `../` |
| 5 | HTML实体编码 | `&#\d+;|&#x[0-9A-Fa-f]+;` | html_entity_decode | `&#46;&#46;&#47;` → `../` |
| 6 | Base64 | `/^[A-Za-z0-9+/]+=*$/` 且len%4==0且len>=8 | base64_decode | `Li4v` → `../` |
| 7 | 混合编码 | 多种编码特征组合 | 逐层解码 | `URL+Unicode` |

**检测流程：**
1. 从请求体提取URL路径或参数值
2. 按优先级依次检测编码特征
3. 如果匹配，立即进行解码
4. 对解码后的内容再次检测爬虫特征
5. 循环步骤2-4，最多10层或直到连续3次无编码匹配

## 误报判定场景

### 场景检查优先级

**场景分类说明：**
- **直接验证类**：可通过单次请求响应直接判定，无需额外推断
- **推断类**：需要根据响应特征或行为模式推断，判定置信度相对较低

当多个误报场景特征同时出现时，按以下优先级判定：

| 优先级 | 场景 | 类型 | 判定依据强度 |
|--------|------|------|-------------|
| 1 | 场景1：搜索引擎爬虫 | 直接验证 | 最强：UA明确 |
| 1 | 场景6：社交媒体预览爬虫 | 直接验证 | 最强：UA明确 |
| 2 | 场景2：内部运维/安全工具 | 直接验证 | 最强：IP明确 |
| 2 | 场景8：可用性监控服务 | 直接验证 | 最强：UA明确 |
| 3 | 场景9：浏览器预连接/预加载 | 推断类 | 强：请求特征 |
| 3 | 场景3：SPA静态资源加载 | 推断类 | 强：请求模式 |
| 4 | 场景7：广告平台爬虫 | 直接验证 | 中：UA明确 |
| 4 | 场景11：云服务商业爬虫 | 推断类 | 中：IP+UA综合 |
| 5 | 场景5：缓存预热/CDN回源 | 推断类 | 中：响应头明确 |
| 5 | 场景10：用户行为分析脚本 | 推断类 | 中：路径特征 |
| 6 | 场景4：移动端App/第三方API | 推断类 | 中：UA特征 |
| 7 | 场景12：内部业务定时任务 | 推断类 | 中：时间规律 |
| 8 | 场景13：反向代理/NAT误报 | 推断类 | 弱：架构推断 |
| 9 | 场景14：间歇性突发流量误报 | 推断类 | 弱：流量模式 |

**冲突处理规则：**
- 如果同时匹配多个场景，选择优先级最高的场景
- 如果优先级相同，按以下标准选择特征更加明确的场景：
  - User-Agent明确 > IP地址推断 > 行为模式推断
  - 官方标识明确 > 自定义标识 > 推断判断
- 对于明显匹配多个场景的情况，标记为"误报（多场景匹配）"并列出所有匹配场景
- 推断类场景的判定结果置信度建议标记为"中"，并在报告中说明推断依据

### 场景1：搜索引擎爬虫

**特征识别：**
- User-Agent包含搜索引擎爬虫标识
- 包括Googlebot、Baiduspider、bingbot、yandexbot等

**误报判定依据：**
1. User-Agent识别：包含官方爬虫标识
2. IP地址验证：来自官方IP段（可选增强）
3. 请求行为：符合正常爬取模式

**搜索引擎爬虫特征：**

| 爬虫类型 | User-Agent特征 | 官方IP段 |
|---------|---------------|---------|
| Google | Googlebot/ | Google公开IP段 |
| 百度 | Baiduspider/ | 百度公开IP段 |
| 微软 | Bingbot/ | Microsoft公开IP段 |
| Yandex | YandexBot/ | Yandex公开IP段 |
| 搜狗 | Sogou web spider/ | 搜狗公开IP段 |
| 360 | 360Spider/ | 360公开IP段 |

**典型示例：**
```
请求: GET /page1
User-Agent: Mozilla/5.0 (compatible; Googlebot/2.1; +http://www.google.com/bot.html)
响应码: 200 OK
说明: Google爬虫正常爬取
```

**误报原因：**
这些合法爬虫通常具有高频访问、遍历目录的特征，极易触发WAF/IPS中的"防扫描"、"目录枚举"或"CC攻击"规则。如果未将这些已知的官方爬虫IP段加入白名单，就会被误判为恶意的信息窃取行为。

**判定逻辑（置信度：高）：**
```
IF User-Agent包含官方爬虫标识（Googlebot、Baiduspider、bingbot、yandexbot等）
AND 请求行为符合正常爬取模式
THEN 误报 = True
```

### 场景2：内部运维/安全工具

**特征识别：**
- 源IP为内部监控段（10.x.x.x、192.168.x.x等）
- User-Agent包含Nessus、Zabbix、Prometheus、AWVS等

**误报判定依据：**
1. IP地址识别：来自内部监控段
2. User-Agent识别：包含运维/安全工具标识
3. 请求行为：符合监控或扫描特征

**内部运维/安全工具特征：**

| 工具类型 | User-Agent特征 | 用途 |
|---------|---------------|------|
| Nessus | Nessus/ | 漏洞扫描 |
| Zabbix | Zabbix Sender | 监控告警 |
| Prometheus | Prometheus/ | 监控数据采集 |
| AWVS | Acunetix Web Vulnerability Scanner | Web漏洞扫描 |
| OpenVAS | OpenVAS | 漏洞扫描 |
| Burp Suite | Burp | 安全测试 |
| Nmap | Nmap | 端口扫描 |

**典型示例：**
```
请求: GET /api/healthcheck
源IP: 10.0.0.5（内部监控段）
User-Agent: Zabbix Sender v5.4.4
响应码: 200 OK
说明: Zabbix监控探测，非攻击行为
```

**误报原因：**
自动化运维脚本和漏扫工具的行为模式（高频请求、探测敏感路径、携带特定Payload）与黑客的攻击前侦察高度相似。如果没有将内部监控IP或漏扫引擎的指纹特征做放行处理，会产生大量"内鬼攻击"的误报。

**判定逻辑（置信度：高）：**
```
IF 源IP为内部监控段（10.x.x.x、192.168.x.x、172.16-31.x.x等）
AND User-Agent包含运维/安全工具标识
AND 请求行为符合监控或扫描特征
THEN 误报 = True
```

### 场景3：SPA静态资源加载

**特征识别：**
- 短时间内同时请求多个.js/.css/.json文件
- Referer为同源页面
- 请求路径通常为/static、/assets等

**误报判定依据：**
1. 请求模式：短时间内并发请求多个静态资源
2. Referer同源：所有请求来自同一页面
3. 资源类型：前端静态资源

**SPA静态资源特征：**

| 资源类型 | 请求特征 | 说明 |
|---------|---------|------|
| JavaScript | .js文件 | 前端脚本 |
| CSS | .css文件 | 样式文件 |
| JSON | .json文件 | 数据文件 |
| 图片 | .png、.jpg等 | 图片资源 |
| 字体 | .woff、.ttf等 | 字体文件 |

**典型示例：**
```
时间顺序：
请求1: GET /static/app.js
请求2: GET /static/style.css
请求3: GET /static/data.json
请求4: GET /static/icon.png
Referer: https://example.com/page

所有请求: User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Chrome/120.0.0.0

说明: SPA静态资源正常加载，非爬虫行为
```

**误报原因：**
这种短时间内对多个文件的高频并发请求，在传统防护设备看来，非常像是一个自动化脚本在进行"敏感文件探测"或"目录遍历"。实际上，这只是前端框架正常的业务初始化行为。

**判定逻辑（置信度：中）：**
```
IF 短时间内（<5秒）同时请求多个静态资源文件（.js、.css、.json等）
AND 所有请求的Referer为同源页面
AND User-Agent为浏览器标识
AND 请求路径符合静态资源模式（/static、/assets等）
THEN 误报 = True
```

### 场景4：移动端App/第三方API

**特征识别：**
- User-Agent包含移动端标识（iOS、Android）
- 请求频率与用户行为相关
- 涉及第三方API调用

**误报判定依据：**
1. User-Agent识别：包含移动端标识
2. 请求频率：频率较高但符合用户行为模式
3. 业务场景：App功能或第三方集成

**移动端特征：**

| 移动端类型 | User-Agent特征 | 示例 |
|-----------|---------------|------|
| iOS | Mozilla/5.0 (iPhone; CPU iPhone OS | iOS |
| Android | Mozilla/5.0 (Linux; Android | Android |
| 微信小程序 | MicroMessenger | 微信内置浏览器 |
| 支付宝小程序 | AlipayClient | 支付宝内置 |

**典型示例：**
```
请求: POST /api/sync
User-Agent: Mozilla/5.0 (Linux; Android 13; SM-G971N) OkHttp/5.0.0
源IP: 移动网络IP
响应码: 200 OK
说明: 移动端App正常API调用，非爬虫行为
```

**误报原因：**
移动端的网络环境复杂，可能会因为重试机制导致请求频率瞬间升高；而B2B的业务对接往往涉及大批量数据的自动化传输。这些正常的程序化调用极易被识别为"暴力破解"或"数据爬取"攻击。

**判定逻辑（置信度：中）：**
```
IF User-Agent包含移动端标识（iOS、Android等）
AND 请求频率与用户行为模式一致（非固定周期）
AND 涉及正常业务功能或第三方API调用
THEN 误报 = True
```

### 场景5：缓存预热/CDN回源

**特征识别：**
- 请求频率高且集中
- X-Forwarded-For包含CDN节点IP
- 可能来自源站预热任务

**误报判定依据：**
1. 请求特征：高频集中请求
2. 响应头特征：包含CDN节点IP
3. 时间特征：预热时间段

**缓存预热特征：**

| 预热类型 | 响应头特征 | 说明 |
|---------|-----------|------|
| Cloudflare | CF-Ray、CF-Cache-Status | Cloudflare预热 |
| Akamai | Akamai-Origin-Hop | Akamai预热 |
| AWS CloudFront | X-Amz-Cf-* | CloudFront预热 |
| Fastly | Fastly-* | Fastly预热 |
| 自建CDN | 自定义标识 | 内部CDN预热 |

**典型示例：**
```
请求: GET /api/data
响应头: X-Cache-Status: MISS
响应头: X-Amz-Cf-Id: xxx
请求模式: 短时间内大量相同请求
说明: CDN缓存预热，触发频率告警
```

**误报原因：**
缓存预热本质上就是一种受控的、高频的"自我爬取"行为。如果不提前告知安全团队或将相关出口IP加白，这种行为会被视为典型的DDoS攻击或恶意爬虫。

**判定逻辑（置信度：中）：**
```
IF 请求频率高且集中在特定时间段（预热期）
AND 响应头包含CDN节点IP或缓存相关标识
AND 请求模式为预热特征（源站回源、缓存MISS）
AND 无其他恶意爬虫特征
THEN 误报 = True
```

### 场景6：社交媒体预览爬虫

**特征识别：**
- User-Agent包含社交媒体爬虫标识
- 包括Twitterbot、facebookexternalhit、LinkedInBot、Slackbot、WeChat等

**误报判定依据：**
1. User-Agent识别：包含社交媒体爬虫标识
2. 请求特征：突发性、短时
3. 业务场景：链接预览、内容抓取

**社交媒体爬虫特征：**

| 社交媒体 | User-Agent特征 | 用途 |
|---------|---------------|------|
| Twitter | Twitterbot | 链接预览 |
| Facebook | facebookexternalhit | 内容抓取 |
| LinkedIn | LinkedInBot | 职业信息抓取 |
| Slack | Slackbot | 消息预览 |
| WeChat | MicroMessenger | 小程序爬取 |

**典型示例：**
```
请求: GET /api/page/preview
User-Agent: Twitterbot/1.0
响应码: 200 OK
说明: Twitter链接预览爬虫
```

**误报原因：**
这些爬虫的User-Agent通常包含Twitterbot、facebookexternalhit、LinkedInBot、Slackbot、WeChat等标识。虽然知名，但安全团队可能未及时更新爬虫IP白名单。这些请求通常是高频、短时的突发流量。

**判定逻辑（置信度：高）：**
```
IF User-Agent包含社交媒体爬虫标识（Twitterbot、facebookexternalhit、LinkedInBot、Slackbot、WeChat等）
AND 请求具有突发性、短时特征
AND 无其他恶意爬虫特征
THEN 误报 = True
```

### 场景7：广告平台爬虫

**特征识别：**
- User-Agent包含广告平台爬虫标识
- 包括AdsBot、Baidu-Ads、Adwords等

**误报判定依据：**
1. User-Agent识别：包含广告爬虫标识
2. 请求特征：广告相关资源
3. IP地址：可能来自广告平台IP段

**广告平台爬虫特征：**

| 广告平台 | User-Agent特征 | 用途 |
|---------|---------------|------|
| Google Ads | AdsBot-Google | 广告爬取 |
| 百度 | Baidu-Ads | 百度广告爬取 |
| 腾讯 | AdsBot | 腾讯广告爬取 |
| 字节跳动 | AdsBot | 字节跳动广告爬取 |
| Facebook | facebookexternalhit | 广告爬取 |

**典型示例：**
```
请求: GET /ads/pixel
User-Agent: AdsBot-Google (+http://www.google.com/adsbot.html)
响应码: 200 OK
说明: Google Ads爬取像素数据
```

**误报原因：**
广告爬虫的行为模式与普通搜索引擎爬虫类似，但其User-Agent和IP段可能与主爬虫不同。例如Google的AdsBot-Google、百度的Baidu-Ads。未单独加白会导致误报。

**判定逻辑（置信度：高）：**
```
IF User-Agent包含广告平台爬虫标识（AdsBot、Baidu-Ads、Adwords等）
AND 请求内容与广告相关（像素、脚本、资源）
AND 无其他恶意爬虫特征
THEN 误报 = True
```

### 场景8：可用性监控服务

**特征识别：**
- 固定间隔请求或访问/health端点
- User-Agent包含UptimeRobot、Pingdom等
- 请求路径固定（如/、/health）

**误报判定依据：**
1. 时间特征：固定时间间隔请求
2. 路径特征：固定路径（/、/health、/ping等）
3. User-Agent识别：监控服务标识

**可用性监控服务特征：**

| 监控服务 | User-Agent特征 | 请求路径 |
|---------|---------------|---------|
| UptimeRobot | UptimeRobot/ | / 或自定义 |
| Pingdom | Pingdom | / 或自定义 |
| StatusCake | StatusCake | / 或自定义 |
| Site24x7 | Site24x7/ | / 或自定义 |
| GTmetrix | PTST/ | / 或自定义 |

**典型示例：```
请求: GET /health
User-Agent: UptimeRobot/2.0
响应码: 200 OK
响应体: {"status": "up"}
说明: UptimeRobot健康检查
```

**误报原因：**
这些服务的探测请求具有极强的规律性（固定时间间隔、固定路径如/、/health），且IP段通常公开。安全设备容易将这种周期性请求误判为"低频慢速扫描"或"存活探测"。

**判定逻辑（置信度：高）：**
```
IF User-Agent包含监控服务标识（UptimeRobot、Pingdom、StatusCake、Site24x7、GTmetrix等）
AND 请求具有周期性特征（固定时间间隔、固定路径）
AND 请求路径为健康检查端点（/、/health、/ping等）
THEN 误报 = True
```

### 场景9：浏览器预连接/预加载

**特征识别：**
- 请求与页面加载时间差极小
- 包含Sec-Fetch-*、Purpose:prefetch等头
- 请求方法可能为OPTIONS

**误报判定依据：**
1. 请求时机：与页面加载时间关联
2. 请求头特征：预连接、预加载标识
3. 请求类型：连接预建立、资源预加载

**浏览器预连接特征：**

| 预连接类型 | 请求头特征 | 用途 |
|-----------|-----------|------|
| 预连接 | Sec-Fetch-Dest, Sec-Fetch-Mode | 连接预建立 |
| 预加载 | Purpose: prefetch | 资源预加载 |
| 预渲染 | Purpose: prerender | 页面预渲染 |
| DNS预解析 | dns-prefetch | DNS预解析 |

**典型示例：**
```
请求: GET /resource
Sec-Fetch-Dest: resource
Sec-Fetch-Mode: cors
Purpose: prefetch
时间差: 与页面加载时间<100ms
说明: 浏览器预连接请求
```

**误报原因：**
这些预连接请求在用户尚未主动点击或滚动到相关区域时就会发出，可能被误判为"自动化脚本探测"或"恶意预取"。尤其在用户行为分析类防护规则中容易触发。

**判定逻辑（置信度：中）：**
```
IF 请求时机与页面加载时间紧密相关（<500ms）
AND 响应头包含预连接/预加载标识（Sec-Fetch-*、Purpose: prefetch等）
AND 请求类型为连接预建立或资源预加载
THEN 误报 = True
```

### 场景10：用户行为分析脚本

**特征识别：**
- 请求路径包含/collect、/track、/analyze等行为分析路径
- 携带行为数据（用户ID、事件数据、页面信息）
- 通常是POST请求，异步发送

**误报判定依据：**
1. 路径特征：包含行为分析标识（collect、track、analyze等）
2. 数据特征：携带行为数据而非爬取数据
3. 请求模式：异步、高频、后台发送

**用户行为分析特征：**

| 分析类型 | 路径特征 | 数据特征 |
|---------|---------|---------|
| 行为收集 | /collect、/track | 用户行为数据 |
| 页面分析 | /analyze、/pageview | 页面访问数据 |
| 事件跟踪 | /event、/log | 事件数据 |
| 数据分析 | /stats、/metrics | 统计数据 |

**典型示例：**
```
请求: POST /api/analytics/track
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Chrome/120.0.0.0
请求体: {"userId": "user123", "action": "click", "element": "button"}
响应码: 200 OK
说明: 用户行为分析脚本，非爬虫行为
```

**误报原因：**
这类请求通常是高频、异步、携带特定参数的POST请求。防护设备若不了解其业务属性，可能将其误判为"数据外发"、"撞库"或"异常爬虫"。

**判定逻辑（置信度：中）：**
```
IF 请求路径包含行为分析标识（/collect、/track、/analyze、/event、/stats等）
AND 请求体包含行为数据（用户ID、事件数据、页面信息等）
AND 请求模式为异步、后台发送
AND 无其他恶意爬虫特征
THEN 误报 = True
```

### 场景11：云服务商业爬虫

- 源IP为公有云服务商（AWS、阿里云、腾讯云等）
- User-Agent为合法商业爬虫
- 请求频率和模式符合业务采集

**误报判定依据：**
1. IP地址特征：来自公有云服务商IP段
2. User-Agent识别：合法商业爬虫
3. 请求特征：符合商业数据采集模式

**云服务IP段特征：**

| 云服务商 | IP段特征 | 地理分布 |
|---------|---------|---------|
| AWS | Amazon.com、Amazon Web Services | 全球 |
| 阿里云 | Aliyun Computing | 中国/全球 |
| 腾讯云 | Tencent Cloud | 中国/全球 |
| Google Cloud | Google Cloud | 全球 |
| Microsoft Azure | Microsoft Azure | 全球 |
| Oracle Cloud | Oracle Cloud | 全球 |

**典型示例：**

User-Agent: DataBot/1.0
响应码: 200 OK

```

**误报原因：**
这些IP地址可能被威胁情报标记为"云服务商"或"代理"，信誉评分较低。安全设备可能基于IP信誉直接拦截或告警，但实际上这是合法的商业数据采集行为。

AND 请求频率和模式符合商业数据采集特征

```

### 场景12：内部业务定时任务

**特征识别：**
- 请求时间高度规律
- 源IP为内部网段
- 请求路径和参数固定

**误报判定依据：**
1. 时间特征：固定时间间隔请求
2. IP地址特征：来自内部网段
3. 请求特征：行为模式单一、固定

**内部定时任务特征：**

| 任务类型 | 时间特征 | 路径特征 |
|---------|---------|---------|
| 数据同步 | 每日固定时间 | /api/sync |
| 报表生成 | 每日/每周固定时间 | /api/report |
| 缓存清理 | 每小时固定时间 | /api/cache/clear |
| 备份任务 | 每日固定时间 | /api/backup |
| 监控检查 | 每分钟固定时间 | /api/monitor |

**典型示例：**
```
请求: POST /api/data/sync
源IP: 192.168.1.10（内部网段）
时间: 每日 02:00:00
请求体: {"sync_type": "full", "data": "..."}
响应码: 200 OK
说明: 内部定时数据同步任务
```

**误报原因：**
这些定时任务的请求频率固定、行为模式单一，与"恶意探测"特征重叠。如果安全团队不熟悉所有内部业务系统的调用模式，容易产生误报。

**判定逻辑（置信度：中）：**
```
IF 请求时间高度规律（固定时间间隔，如每天/每小时/每分钟）
AND 源IP为内部网段
AND 请求路径和参数固定
AND 行为模式单一，无爬取特征
THEN 误报 = True
```

### 场景13：反向代理/NAT误报

**特征识别：**
- 多个不同源IP汇聚到同一目标
- X-Forwarded-For包含多个IP地址
- 实际来源IP与源IP字段不一致

**误报判定依据：**
1. 响应头特征：X-Forwarded-For包含多个IP
2. IP地址不一致：实际来源IP与源IP字段不匹配
3. 代理/NAT特征：经过反向代理或NAT转换

**反向代理/NAT特征：**

| 类型 | 响应头特征 | 说明 |
|------|-----------|------|
| 反向代理 | X-Forwarded-For | 代理IP |
| 负载均衡 | X-Forwarded-For | LB节点IP |
| NAT | X-Real-IP | 真实客户端IP |
| CDN | X-Forwarded-For | CDN边缘节点IP |

**典型示例：**
```
请求: GET /api/data
源IP字段: 1.2.3.4
响应头: X-Forwarded-For: 10.0.0.5, 1.2.3.4, 192.168.1.1
说明: 经过反向代理/NAT，源IP字段不准确
```

**误报原因：**
实际爬虫的源IP可能是经过反向代理或NAT转换后的地址。如果安全设备仅根据源IP字段判断，可能会将合法爬虫误判为恶意源，或反之。

**判定逻辑（置信度：低）：**
```
IF 响应头包含多个IP地址（X-Forwarded-For等）
AND 实际来源IP与源IP字段不一致
AND 需要结合X-Forwarded-For中的第一个IP判断
THEN 需重新评估（无法直接判定为误报）
```

### 场景14：间歇性突发流量误报

**特征识别：**
- 突发性高频请求
- 持续时间短
- 之后恢复正常

**误报判定依据：**
1. 流量特征：突发性高频、短时间
2. 时间特征：持续期短
3. 后续状态：恢复正常

**间歇性突发流量特征：**

| 流量类型 | 触发原因 | 持续时间 |
|---------|---------|---------|
| 内容发布 | 热点内容发布后流量激增 | 几分钟到几小时 |
| 营销活动 | 营销活动开始/结束 | 活动期间 |
| 事件触发 | 新闻事件、热点事件 | 事件期间 |
| 推广启动 | 推广活动开始 | 推广期间 |

**典型示例：**
```
时间: 14:00-14:30
请求频率: 突发性高频
源IP: 多个不同IP
后续: 15:00后恢复正常
说明: 可能是热点内容发布导致的合法流量激增
```

**误报原因：**
某些合法业务活动（如内容发布、营销活动、热点事件）会产生突发性的高频流量。这种突发流量模式与恶意爬虫的特征重叠，容易产生误报。

**判定逻辑（置信度：低）：**
```
IF 请求频率突发性升高
AND 持续时间短（通常<1小时）
AND 突发流量前后正常
AND 需结合业务上下文判断（如内容发布、营销活动等）
THEN 误报 = True（需业务确认）
```

## 非误报判定标准

### 真实爬虫告警特征

若不满足上述任何误报场景，且存在以下明确爬虫证据，则判定为**真实告警**：

#### 1. 数据爬取成功

**特征：**
- 爬取大量页面内容或数据
- 响应包含被爬取的完整信息

**判定：**
```
IF 请求频率异常高（超出正常业务范围）
AND 响应包含目标数据内容（如商品信息、文章内容等）
AND 数据为预期之外或敏感信息
AND 无其他误报场景特征
THEN 真实告警 = True
```

#### 2. 恶意目录遍历

**特征：**
- 系统性遍历网站目录结构
- 尝试访问大量隐藏或敏感路径

**判定：**
```
IF 请求包含系统性的目录遍历模式（如/a/、/b/、/c/等）
AND 尝试访问大量隐藏或敏感路径（/admin、/backup、/.git等）
AND 响应包含目录列表或文件内容
THEN 真实告警 = True
```

#### 3. 敏感文件访问

**特征：**
- 尝试访问敏感文件（如/etc/passwd、.env、config.xml）
- 成功获取文件内容

**判定：**
```
IF 请求尝试访问敏感文件路径（/etc/passwd、.env、config.xml、backup.zip等）
AND 响应成功返回文件内容
AND 文件内容超出正常业务范围
THEN 真实告警 = True
```

#### 4. 恶意资源枚举

**特征：**
- 系统性枚举API端点或资源
- 尝试大量不同的参数组合

**判定：**
```
IF 请求包含系统性的API端点或资源枚举模式
AND 尝试大量不同的参数组合
AND 响应包含有效枚举结果
AND 无其他误报场景特征
THEN 真实告警 = True
```

### 使用说明

1. **触发条件**：当用户需要分析网络爬虫类告警时使用此cookbook
2. **分析步骤**：
   - 提取请求URL和响应头
   - 识别User-Agent和IP地址
   - 首先检查高优先级场景（UA明确、IP明确）
   - 按优先级检查误报场景
   - 如无匹配，检查真实爬虫特征
   - 输出判定结果和依据

3. **置信度评估**：
   - 高：明确匹配误报场景（User-Agent明确、IP明确）或真实爬虫特征（数据爬取、目录遍历）
   - 中：部分匹配，需要额外上下文确认或推断类场景
   - 低：无明确特征，需要人工复核

## 注意事项

1. **User-Agent识别优先**：User-Agent是识别爬虫类型的最重要依据，优先检查
2. **IP地址白名单**：维护官方爬虫IP段和内部监控IP段的白名单
3. **行为模式分析**：结合请求频率、时间规律、请求路径综合判断
4. **上下文关联**：结合业务上下文（如内容发布、营销活动）综合判断
5. **误报场景覆盖**：确保覆盖所有14个误报场景
6.  **场景优先级**：高优先级场景优先匹配，低优先级场景需要更多上下文验证
7.  **无法判定处理**：对于无法判定的告警，标记为"待人工复核"并记录原因
8.  **场景冲突处理**：多场景匹配时选择优先级最高的，或标记为"多场景匹配"
9.  **反向代理/NAT影响**：注意X-Forwarded-For等响应头，避免基于错误源IP误判
10. **云服务IP段**：注意公有云服务商IP段可能被误判为代理或恶意源
11. **间歇性流量**：区分恶意突发流量和合法的营销/热点事件流量
12. **内部工具指纹**：识别内部运维和安全工具的User-Agent和指纹特征
