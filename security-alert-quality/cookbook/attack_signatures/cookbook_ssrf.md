## 快速判定摘要

以下为脚本/LLM自动判定使用的结构化规则，详细分析见后续章节。

### 真阳性指标

| 置信度 | 特征 | 说明 |
|--------|------|------|
| 高 | 请求URL参数含 `http://169.254.169.254/` 或 `metadata.google.internal` | 云元数据服务探测（AWS/GCP） |
| 高 | 请求URL参数含 `http://127.0.0.1/` 或 `http://localhost/` | 内网服务探测 |
| 高 | 请求URL参数含 `file:///` / `gopher://` / `dict://` | 危险协议利用 |
| 高 | 请求URL参数含内网IP段 `10.`, `172.16-31.`, `192.168.` | SSRF内网探测 |
| 中 | URL参数值为完整URL且指向上游内部服务 | 可能的SSRF代理 |
| 中 | 响应体返回了内网服务的响应内容 | SSRF成功回显 |

### 误报指标

| 置信度 | 特征 | 说明 |
|--------|------|------|
| 高 | URL参数值为公网合法URL（如回调地址、Webhook） | 正常的Webhook/回调 |
| 高 | OAuth redirect_uri 参数 | 正常OAuth流程 |
| 中 | 图片代理/URL预览服务获取外部URL | 正常业务功能 |
| 中 | CDN回源请求 | 基础设施请求 |

### 判定优先级

1. 先查目标是否为云元数据地址（169.254.169.254） → 高置信度SSRF探测
2. 再查是否使用危险协议（file/gopher/dict） → 高置信度SSRF
3. 最后查是否为内网IP → 需结合业务上下文判断

## 技能概述

网络安全设备通过内置规则检测SSRF注入攻击，但由于规则配置、检测逻辑、业务场景、网络架构等因素，容易产生大量误报。本cookbook提供系统化的SSRF注入误报识别方法，帮助分析人员准确判定告警真实性。

## 角色定位

你是一名专业的安全告警研判分析师，专注于SSRF类告警的误报识别与判定。

## 分析输入

### 数据来源

从网络安全设备告警日志CSV中提取以下关键字段：

| 字段 | 用途 | 说明 |
|------|------|------|
| 请求体 | 分析URL参数 | 包含用户可控的URL参数 |
| 响应体 | 判断请求结果 | 包含服务器返回内容 |
| 响应码 | 判断请求状态 | 200/302/403/404/500等状态码分析 |
| 响应头 | 分析拦截机制 | WAF头、重定向头、CSP等 |
| 载荷 | 分析流量特征 | 网络流量中的原始数据 |

### 编码绕过识别

在分析请求体时，必须检测以下编码方式（按检测优先级排序）：

| 优先级 | 编码类型 | 检测规则 | 解码方法 | 示例 |
|--------|----------|----------|----------|------|
| 1 | URL编码 | `%[0-9A-Fa-f]{2}` | urldecode | `%68%74%74%70%3A%2F%2F` → `http://` |
| 2 | Base64 | `/^[A-Za-z0-9+/]+=*$/` 且len%4==0 | base64_decode | `aHR0cDovL2xvY2FsaG9zdC8=` → `http://localhost/` |
| 3 | Hex编码 | `\\x[0-9A-Fa-f]{2}` | hex2bin | `\x68\x74\x74\x70\x3A\x2F\x2F` → `http://` |
| 4 | Unicode编码 | `\\u[0-9A-Fa-f]{4}` | unicode_decode | `\u0068\u0074\u0074\u0070` → `http` |
| 5 | HTML实体编码 | `&#\d+;|&#x[0-9A-Fa-f]+;` | html_entity_decode | `&#x68;&#x74;&#x74;&#x70;` → `http` |
| 6 | 双重/多重编码 | 多层编码特征 | 多次解码 | `%2568%2574` → `%68%74` → `h`t` |
| 7 | 混淆编码 | 各种组合编码 | 逐层解码 | 混合URL和Base64 |

**检测流程：**
1. 从请求体提取URL参数值
2. 按优先级依次检测编码特征
3. 如果匹配，进行解码
4. 记录已解码内容，如解码结果与之前相同则停止（防止解码循环）
5. 对解码后的内容再次检测（处理多重编码）
6. 最多解码10层或直到内容不再变化

### 场景1：URL完全固定/可信配置

**特征识别：**
- URL无用户输入变量
- URL硬编码或从可信配置读取

**误报判定依据：**
1. URL分析：所有组成部分都不受用户输入控制
2. 配置来源：URL来自配置文件、环境变量等可信来源
3. 无动态拼接：无字符串拼接或模板变量

**固定URL特征：**

| 特征类型 | 示例 | 说明 |
|----------|------|------|
| 硬编码URL | `https://api.example.com/endpoint` | 代码中固定 |
| 配置文件 | 从config读取 | 可信配置源 |
| 环境变量 | 从env读取 | 可信配置源 |
| 白名单域名 | 从白名单选择 | 可信域名列表 |

**典型示例：**
```
URL: https://api.payment.com/callback
说明: URL硬编码或从可信配置读取，无用户输入
```

**判定逻辑（置信度：高）：**
```
IF URL的所有组成部分都不受用户输入控制
AND URL来自可信配置源（硬编码、配置文件、环境变量等）
AND 无动态字符串拼接或模板变量
THEN 误报 = True
```

### 场景2：白名单校验+协议限制

**特征识别：**
- 响应为业务错误页（非200或200但含"非法URL"）
- 扫描器未深入分析响应内容

**误报判定依据：**
1. 响应码：非200或200但返回错误信息
2. 响应内容：包含"非法URL"、"不允许"、"白名单"等关键词
3. 校验机制：URL被白名单校验或协议限制

**拦截响应特征：**

| 响应特征 | 示例 | 说明 |
|----------|------|------|
| HTTP状态码 | 400/403/404 | 请求被拒绝 |
| 错误信息 | "非法URL"、"不允许的域名" | 白名单拦截 |
| 协议限制 | "仅支持HTTP/HTTPS" | 协议拦截 |
| 格式验证 | "URL格式错误" | 格式验证失败 |

**典型示例：**
```
请求: POST /api/fetch?url=file:///etc/passwd
响应: {"error": "不允许的URL，仅支持HTTP/HTTPS协议"}
说明: 协议限制拦截，非真实SSRF
```

**判定逻辑（置信度：高）：**
```
IF 响应码为400/403/404或响应成功但返回错误信息
AND 响应内容包含"非法URL"、"不允许"、"白名单"、"仅支持HTTP/HTTPS"等关键词
THEN 误报 = True
```

### 场景3：云环境/容器隔离

**特征识别：**
- 云元数据地址请求无响应
- K8s网络策略阻断
- 安全组规则限制

**误报判定依据：**
1. 目标地址：云元数据地址（如169.254.169.254）或内网IP
2. 网络隔离：请求被基础设施阻断
3. 无响应：请求超时或连接被拒绝

**云环境隔离特征：**

| 隔离类型 | 特征 | 示例 |
|----------|------|------|
| 元数据服务 | 169.254.169.254 | 云厂商限制访问 |
| K8s网络策略 | Pod间网络隔离 | Service隔离 |
| 安全组规则 | 端口/IP限制 | 云安全组 |
| VPC隔离 | 内网VPC隔离 | VPC边界 |

**典型示例：**
```
请求: GET /api/fetch?url=http://169.254.169.254/latest/meta-data/
响应: 超时/连接被拒绝
说明: 云平台元数据地址被网络隔离阻断
```

**判定逻辑（置信度：中）：**
```
IF 请求目标为云元数据地址（169.254.169.254）或内网地址
AND 请求超时或连接被拒绝
AND 响应无敏感信息泄露
THEN 误报 = True
```

### 场景4：WAF/中间件前置拦截

**特征识别：**
- 403响应来自WAF
- 响应头包含WAF标识

**误报判定依据：**
1. 响应码：403 Forbidden
2. WAF标识：响应头包含WAF特征
3. 拦截位置：请求被边缘节点拦截，未到达源站

**WAF响应头特征：**

| WAF类型 | 响应头特征 | 示例 |
|----------|-----------|------|
| Cloudflare | CF-Ray、CF-Cache-Status | CF-Ray: 1234567890 |
| Sucuri | X-Sucuri-ID、X-Sucuri-Cache | X-Sucuri-ID: S123456 |
| AWS WAF | x-amz-waf-* | x-amz-waf-action: BLOCK |
| Akamai | Akamai-* | Akamai-Origin-Hop |
| Fastly | Fastly-* | Fastly-Request-ID |

**典型示例：**
```
请求: GET /api/fetch?url=http://internal.service/api
响应: 403 Forbidden
响应头: CF-Ray: 1234567890
说明: 请求被Cloudflare WAF拦截
```

**判定逻辑（置信度：高）：**
```
IF 响应码为403或类似拦截状态码
AND 响应头包含WAF特征标识（CF-Ray、X-Sucuri-ID、x-amz-waf等）
AND 请求未到达源站
THEN 误报 = True
```

### 场景5：盲注无OOB证据

**特征识别：**
- 无DNS/HTTP回调记录
- 仅依赖响应时间或错误码

**误报判定依据：**
1. 响应特征：仅响应时间差异或HTTP 500错误
2. OOB证据缺失：无带外通道（DNS/HTTP）回调
3. 网络原因：可能为网络抖动或超时

**盲注类型特征：**

| 盲注类型 | 特征 | 误报原因 |
|----------|------|---------|
| 时间盲注 | 响应时间差异 | 网络抖动 |
| 错误盲注 | HTTP 500错误 | 业务超时 |
| 状态盲注 | 状态码变化 | 正常业务行为 |

**典型示例：**
```
请求: GET /api/fetch?url=http://internal.service/api
响应: 500 Internal Server Error（响应时间2000ms）
请求: GET /api/fetch?url=http://safe.service/api
响应: 200 OK（响应时间1800ms）
说明: 响应时间差异可能来自网络抖动，无OOB证据支持
```

**判定逻辑（置信度：中）：**
```
IF 仅有响应时间差异或HTTP 500错误
AND 无带外通道证据（DNS解析记录、HTTP回调、外部服务等）
AND 响应内容无敏感信息
AND Payload无明显盲注攻击特征（如SQL盲注的时间间隔、布尔盲注的明显条件差异）
THEN 误报 = True
```

### 场景6：重定向跟随误判

**特征识别：**
- 响应状态码302/301
- Location头指向内网地址
- 需确认后端客户端是否允许重定向

**误报判定依据：**
1. 响应码：302/301/307/308
2. Location头：指向内网地址
3. 客户端限制：后端客户端不跟随或限制重定向

**HTTP客户端重定向特征：**

| 客户端类型 | 重定向行为 | 限制配置 |
|-----------|-----------|---------|
| Java HttpURLConnection | 默认跟随 | 可配置maxRedirects |
| Python requests | 默认跟随 | 可配置allow_redirects |
| Python urllib3 | 默认不跟随 | 需显式处理 |
| Node.js http | 默认不跟随 | 需手动处理 |
| curl | 默认跟随 | 可配置--max-redirs |

**典型示例：**
```
请求: POST /api/fetch
请求体: {"url": "http://internal.service/api"}
响应: 302 Found
Location: http://internal.service/api
说明: 如果后端客户端不跟随重定向或限制为同域，攻击无效
```

**判定逻辑（置信度：中）：**
```
IF 响应状态码为302/301/307/308
AND Location头指向内网地址
AND 后端客户端不跟随重定向或限制为同域
AND 客户端无法执行跨域重定向
THEN 误报 = True
```

### 场景7：URL解析差异

**特征识别：**
- Payload涉及URL认证信息（@）、反斜杠（\）、异常编码
- 不同解析器结果不同

**误报判定依据：**
1. Payload特征：包含解析差异技巧
2. 解析器差异：扫描器与后端使用不同解析器
3. 拦截验证：后端解析器正确拦截

**解析差异技巧：**

| 技巧 | 示例 | 受影响解析器 |
|------|------|-------------|
| URL认证(@) | `http://attacker@internal.service/api` | 某些旧版解析器 |
| 反斜杠(\) | `http://internal.service/` | Windows系统 |

```
THEN 误报 = True
```

## 非误报判定标准

### 真实SSRF告警特征

若不满足上述任何误报场景，且存在以下明确SSRF漏洞特征，则判定为**真实告警**：

#### 1. 内网服务探测成功

**特征：**
- 成功访问内网地址
- 返回内网服务信息

**判定：**
```
IF 请求成功访问内网地址（非localhost）
AND 响应返回内网服务信息
AND 内网服务不是公开服务
AND 无明显的白名单校验或拦截机制
THEN 真实告警 = True
```

#### 2. 云元数据泄露

**特征：**
- 成功访问云元数据地址
- 返回敏感元数据

**判定：**
```
IF 请求成功访问云元数据地址（169.254.169.254）
AND 响应返回敏感元数据（IAM凭证、实例ID、私钥等）
AND 云平台未正确配置元数据访问限制
THEN 真实告警 = True
```

#### 3. 端口扫描成功

**特征：**
- 内网端口扫描成功
- 返回服务响应

**判定：**
```
IF 请求成功扫描内网端口
AND 响应返回服务信息或版本信息
AND 端口非公开服务
THEN 真实告警 = True
```

#### 4. 带外通道回连

**特征：**
- 建立带外连接
- 接收外部服务响应

**判定：**
```
IF 建立带外连接（DNS解析、HTTP回调等）
AND 接收到外部服务响应
AND 响应包含敏感信息
THEN 真实告警 = True
```

#### 5. 内网敏感服务访问

**特征：**
- 访问内网敏感服务
- 获取敏感数据

**判定：**
```
IF 成功访问内网敏感服务（数据库、管理后台、配置中心等）
AND 获取敏感数据（数据库内容、配置信息、凭证等）
THEN 真实告警 = True
```

### 使用说明

1. **触发条件**：当用户需要分析SSRF注入类告警时使用此cookbook
2. **分析步骤**：
   - 提取请求体和响应体
   - 识别并解码编码内容
   - 首先检查高优先级场景（URL固定、业务明确）
   - 按优先级检查误报场景
   - 如无匹配，检查真实SSRF特征
   - 输出判定结果和依据

3. **置信度评估**：
   - 高：明确匹配误报场景（URL固定、业务明确、响应明确）或真实SSRF特征（有敏感信息泄露）
   - 中：部分匹配，需要额外上下文确认或推断类场景
   - 低：无明确特征，需要人工复核

## 注意事项

1. **编码识别优先级**：按URL编码 → Base64 → Hex → Unicode → HTML实体 → 多重编码的顺序检测
2. **编码检测流程**：最多解码10层或直到内容不再变化，防止解码循环
3. **URL分析优先**：首先分析URL是否受用户输入控制
4. **响应分析**：重点分析响应内容是否包含敏感信息
5. **业务上下文**：结合URL路径、参数名、业务场景综合判断
6. **误报场景覆盖**：确保覆盖所有22个误报场景
7. **场景优先级**：高优先级场景优先匹配，低优先级场景需要更多上下文验证
8. **无法判定处理**：对于无法判定的告警，标记为"待人工复核"并记录原因
9. **场景冲突处理**：多场景匹配时选择优先级最高的，或标记为"多场景匹配"
10. **云环境特征**：注意云原生环境（K8s、Docker、容器）的网络隔离特性
11. **协议限制**：注意后端客户端的协议和方法限制
12. **业务功能识别**：区分正常业务功能（Webhook、第三方集成）和恶意SSRF攻击
13. **OOB证据**：带外通道证据是判定真实SSRF的重要依据
14. **风险等级评估**：响应无敏感信息的低风险误报与高风险误报的区别
