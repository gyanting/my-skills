## 快速判定摘要

以下为脚本/LLM自动判定使用的结构化规则，详细分析见后续章节。

### 真阳性指标

| 置信度 | 特征 | 说明 |
|--------|------|------|
| 高 | 请求含 `UNION SELECT` + `information_schema` 或 `FROM` | SQL联合查询注入 |
| 高 | 请求含 `' OR '1'='1` / `" OR "1"="1` / `OR 1=1--` | SQL布尔盲注 |
| 高 | 请求含 `SLEEP(`, `BENCHMARK(`, `WAITFOR DELAY`, `pg_sleep` | SQL时间盲注 |
| 高 | 请求含 `LOAD_FILE(`, `INTO OUTFILE`, `xp_cmdshell` | SQL文件操作/命令执行 |
| 高 | 请求含 `0x[0-9a-fA-F]{8,}` 十六进制SQL payload | 十六进制编码绕过 |
| 中 | 请求含 `CHAR(`, `CONCAT(`, `GROUP_CONCAT` | SQL函数调用 |
| 中 | URL参数含 `'` 或 `"` 且响应有 SQL 错误 | 引号注入+错误回显 |
| 中 | 响应体含 `SQL syntax`, `mysql_fetch`, `ORA-`, `PostgreSQL` 错误 | 数据库错误泄露 |

### 误报指标

| 置信度 | 特征 | 说明 |
|--------|------|------|
| 高 | 搜索框中输入含 `'` 或 `OR` 的普通文本 | 正常用户搜索行为 |
| 高 | JSON/XML数据中的SQL关键字作为字面量 | 结构化数据传输 |
| 中 | API文档/Swagger页面中的SQL示例 | 开发文档页面 |
| 中 | SQL关键字出现在正常的数据库管理工具请求中 | DBA运维工具 |
| 低 | URL参数含SQL关键字但经参数化查询处理 | 后端安全防护 |

### 判定优先级

1. 先查 `UNION SELECT` + `information_schema` → 几乎是肯定的注入探测
2. 再查盲注特征（`' OR '1'='1`, `SLEEP(`） → 配合响应差异判定
3. 检查响应体是否包含数据库错误 → 确认注入成功
4. 区分搜索引擎搜索SQL关键词 vs 实际SQL注入payload

## 技能概述

网络安全设备通过内置规则检测SQL注入攻击，但由于规则配置、检测逻辑、业务场景等因素，容易产生大量误报。本cookbook提供系统化的SQL注入误报识别方法，帮助分析人员准确判定告警真实性。

## 角色定位

你是一名专业的安全告警研判分析师，专注于SQL注入类告警的误报识别与判定。

## 分析输入

### 数据来源

从网络安全设备告警日志CSV中提取以下关键字段：

| 字段 | 用途 | 说明 |
|------|------|------|
| 请求体 | 分析注入Payload | 包含用户输入的参数内容 |
| 响应体 | 判断注入结果 | 包含服务器返回内容，用于验证注入是否成功 |
| 响应码 | 判断请求状态 | 200/404/500等状态码分析 |
| 载荷 | 分析流量特征 | 网络流量中的原始数据 |

### 编码绕过识别

在分析请求体时，必须检测以下编码方式（按检测优先级排序）：

| 优先级 | 编码类型 | 检测规则 | 解码方法 | 示例 |
|--------|----------|----------|----------|------|
| 1 | URL编码 | `/%[0-9A-Fa-f]{2}/` | urldecode | `%27` → `'` |
| 2 | Base64 | `/^[A-Za-z0-9+/]+=*$/` 且len%4==0 | base64_decode | `JyBPUg==` → `' OR` |
| 3 | Hex编码 | `/^0x[0-9A-Fa-f]+$/` 或连续十六进制对 | hex2bin | `0x27` → `'` |
| 4 | Unicode编码 | `\\u[0-9A-Fa-f]{4}` | unicode_decode | `\u0027` → `'` |
| 5 | HTML实体编码 | `/&#\d+;|&#x[0-9A-Fa-f]+;/` | html_entity_decode | `&#39;` → `'` |
| 6 | ASCII编码 | `/CHAR\(\d+\)/i` 或 `/ASCII\(\d+\)/i` | chr函数 | `CHAR(39)` → `'` |
| 7 | JSON编码 | `\\["'\\\/]` 等转义字符 | JSON解析 | `\"` → `"` |
| 8 | UTF-7编码 | 以`+`开头、`-`结尾的特征字符集 | UTF-7解码 | `+ACs-` → `'` |
| 9 | 双重/多重编码 | 多层编码特征 | 多次解码 | `%2527` → `%27` → `'` |

**检测流程：**
1. 从请求体提取参数值
2. 按优先级依次检测编码特征
3. 如果匹配，进行解码
4. 对解码后的内容再次检测（处理多重编码）
5. 最多解码3层，防止解码循环

**编码检测优先级说明：**
- URL编码优先：Web应用中最常见的编码方式，也是注入绕过的首选
- Base64次之：常用于参数加密传输，解码后可能触发注入规则
- Hex编码：常见于SQL注入绕过，特别是注释后的十六进制字符串
- UTF-7编码：较少见但可能被用于绕过某些过滤机制

当多个误报场景特征同时出现时，按以下优先级判定：

| 优先级 | 场景 | 类型 | 判定依据强度 |
|--------|------|------|-------------|

| 8 | 场景13：参数化占位符 | 直接验证 | 中：占位符语法明确 |
| 9 | 场景14：Base64业务数据 | 直接验证 | 中：编码类型明确 |

| 12 | 场景19：代码注释SQL语句 | 直接验证 | 中：注释标记明确 |

- 如果同时匹配多个场景，选择优先级最高的场景

### 场景1：请求参数包含特殊字符但无数据库交互

**特征识别：**
- 参数值包含单引号（`'`）、双引号（`"`）、反斜杠（`\`）
- 参数值包含注释符（`--`、`#`、`/**/`）
- 参数值包含分号（`;`）

**误报判定依据：**
1. 检查请求体：参数值包含特殊字符
2. 检查响应体：无数据库错误信息，返回正常业务内容
3. 上下文分析：参数仅用于日志记录、JSON处理、文件名等非SQL场景

**典型Payload示例：**
```
参数名: filename
参数值: test'20240521.log
```

**判定逻辑：**
```
IF 参数值包含SQL特殊字符
AND 响应体无数据库错误
AND 参数未被用于SQL查询上下文
THEN 误报 = True
```

### 场景2：SQL关键字在业务数据中触发规则

**特征识别：**
- 参数值包含SQL关键字（`SELECT`、`AND`、`OR`、`UNION`等）
- 响应体原样返回参数内容

**误报判定依据：**
1. 检查请求体：参数值包含SQL关键字
2. 检查响应体：SQL关键字出现在正常业务数据中（如评论内容、文章标题）
3. 无注入行为特征：无SQL语法结构，仅为关键词出现

**典型示例：**
```
请求: POST /api/comment
请求体: {"content": "请选择select颜色，或者or其他选项"}
响应: {"msg": "评论成功：请选择select颜色，或者or其他选项"}
```

**判定逻辑：**
```
IF 请求参数包含SQL关键字
AND 响应体原样显示该内容
AND 无SQL语法结构（无WHERE、ORDER BY等）
AND 业务上下文允许自由文本输入
THEN 误报 = True
```

### 场景3：后端返回数据库错误信息，但非注入引起

**特征识别：**
- 响应体包含数据库错误信息
- 错误类型为业务异常或连接问题

**误报判定依据：**
1. 检查响应体：包含数据库错误
2. 错误类型识别：
3. 上下文关联：错误信息与注入Payload无直接关联

| 错误类型 | 典型错误信息 | 误报原因 | 上下文验证 |
|----------|-------------|---------|-----------|
| 连接错误 | `MySQL server has gone away` | 数据库连接断开 | 任何参数都会触发，非注入特有 |
| 唯一键冲突 | `Duplicate entry 'xxx' for key 'PRIMARY'` | 业务逻辑重复提交 | 需检查Payload是否包含插入操作 |
| 数据超长 | `Data too long for column 'xxx'` | 字段长度限制 | 检查Payload长度是否异常 |
| 字段不存在 | `Unknown column 'xxx' in 'field list'` | 表结构变更未同步 | 检查Payload是否试图引用不存在字段 |
| 类型不匹配 | `Incorrect integer value: 'xxx'` | 参数类型转换失败 | 检查Payload是否包含类型混淆Payload |
| 超时错误 | `Lock wait timeout exceeded` | 并发锁等待超时 | 与注入Payload无关，属于并发问题 |
| 语法错误 | `You have an error in your SQL syntax` | 需进一步分析 | 需检查是否由注入Payload触发 |

**重要说明：**
- 对于语法错误，需结合Payload上下文判断。如果Payload包含明显的SQL注入特征且错误信息显示Payload部分内容，可能为真实注入
- 对于唯一键冲突、类型不匹配等错误，如果Payload正常无异常，则判定为误报

**判定逻辑：**
```
IF 响应体包含数据库错误
AND (错误类型为连接错误/超时错误
     OR (错误类型为业务错误 AND Payload无注入特征))
AND 错误信息不包含注入Payload的回显（语法错误类需特殊判断）
THEN 误报 = True
```

### 场景4：动态表名/列名/排序参数

**特征识别：**
- 参数用于动态排序（`ORDER BY`）
- 参数值允许枚举值（如`id`、`name`、`create_time`）

**误报判定依据：**
1. 检查请求：参数值为合法枚举值
2. 检测工具行为：工具发送异常Payload（如`id AND 1=1`）
3. 响应分析：服务器返回404/400或参数验证错误，非注入成功

**典型示例：**
```
正常请求: GET /api/list?sort=id
检测工具: GET /api/list?sort=id+AND+1=1
响应: {"error": "非法排序字段"}
```

**判定逻辑：**
```
IF 参数值为合法枚举值
AND 检测工具发送异常Payload
AND 服务器拒绝非法值（参数校验）
AND 无SQL语法错误回显
THEN 误报 = True
```

### 场景5：输入经过编码或应用采用安全处理方式

**特征识别：**
- 请求参数经过编码
- 响应无注入成功迹象

**误报判定依据：**
1. 检查请求体：识别编码类型（见上文编码绕过识别）
2. 解码分析：解码后的内容包含注入特征
3. 响应验证：应用未执行注入或已做安全处理

**安全处理方式识别：**

| 处理方式 | 可识别特征 | 判定依据 |
|----------|-----------|---------|
| 参数化查询 | 响应正常，无SQL错误 | 特殊字符不触发SQL语法错误 |
| 预编译语句 | 响应正常，查询成功 | SQL与参数分离，特殊字符安全 |
| ORM框架 | 响应正常，无异常 | 如Hibernate、MyBatis自动参数化 |
| 输入过滤/转义 | 特殊字符被处理 | 响应显示参数已被转义或过滤 |

**注意：** 从单次请求响应无法直接判断后端使用何种技术，需通过响应特征间接推断。

**判定逻辑：**
```
IF 请求参数经过编码
AND 解码后包含注入特征
AND 响应体无SQL语法错误
AND 响应体无注入成功迹象（无额外数据返回）
AND 响应正常（200）或返回预期业务结果
THEN 误报 = True
```

### 场景6：WAF或扫描器规则过于宽泛

**特征识别：**
- 检测规则使用简单正则表达式
- 正则表达式匹配过于宽松

**误报判定依据：**
1. 规则分析：检测规则包含宽泛匹配模式
2. 上下文验证：匹配内容在业务中合理存在

**典型宽泛规则：**

| 规则示例 | 匹配内容 | 误报场景 |
|----------|---------|---------|
| `/\b(SELECT|UNION)\b/i` | 包含关键字 | 评论"请选择select颜色" |
| `/\'\s+OR\s+\d+\s*=\s*\d+/i` | 单引号+OR+数字 | ID=1或2正常业务参数 |
| `/\bAND\b/i` | AND关键字 | 逻辑表达"true and false" |
| `/\-\-.*/` | 双横线 | 范围表示"价格范围100--200" |
| `/\w+\s*=\s*\w+/i` | 等号表达式 | "count=10"正常参数 |

**判定逻辑：**
```
IF 检测规则为宽泛正则表达式
AND 匹配内容在业务上下文中合理
AND 无注入意图和SQL语法结构
AND 参数未被用于SQL查询
THEN 误报 = True
```

### 场景7：响应中存在"调试模式"或堆栈信息

**特征识别：**
- 应用开启调试模式
- 响应体包含详细异常堆栈

**误报判定依据：**
1. 检查响应体：包含调试信息和SQL片段
2. 堆栈分析：SQL片段来自代码上下文，非注入回显
3. 对比请求：请求参数未影响SQL语句结构

**典型示例：**
```
响应体:
```
Traceback (most recent call last):
  File "app.py", line 123, in get_user
    cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))
...
```
```

**判定逻辑：**
```
IF 响应体包含调试堆栈
AND 堆栈中的SQL语句来自代码上下文
AND SQL语句结构未受请求参数影响
AND 无注入Payload的回显
THEN 误报 = True
```

### 场景8：并发或状态依赖造成响应错位

**特征识别：**
- 高并发或测试环境
- 响应内容与请求不匹配

**误报判定依据：**
1. 时间戳分析：响应时间异常
2. 请求响应关联性：响应内容与当前请求逻辑不符
3. 会话关联：响应对应其他请求的会话

**判定逻辑：**
```
IF 响应内容与当前请求逻辑不匹配
AND 时间戳显示响应异常延迟
AND 无法建立请求-响应的因果关联
THEN 误报 = True
```

### 场景9：框架/语言层自动输入处理

**特征识别：**
- 特殊字符被框架或语言层自动处理
- 处理后的内容不构成注入

**与场景10的区别：**
- **场景9**：处理逻辑来自框架/语言层自动机制（如PHP的magic_quotes、数据库驱动的自动转义、ORM框架的参数化）
- **场景10**：处理逻辑来自应用开发者自定义的过滤/验证函数（如开发者编写的sanitize函数、过滤器中间件）

**误报判定依据：**
1. 输入处理方式识别：

| 处理方式 | 特征 | 判定依据 |
|----------|------|---------|
| PHP Magic Quotes | 单引号/双引号自动加反斜杠 | 历史版本PHP特性 |
| 数据库驱动转义 | 特殊字符自动转义 | 如PDO、mysqli的自动转义 |
| ORM自动处理 | 参数自动绑定 | Hibernate、MyBatis等 |
| 字符集处理 | 多字节字符处理影响 | 如GBK编码的宽字节注入防护 |

2. 响应验证：参数值特殊但响应正常

**典型示例：**
```
输入: 1' OR '1'='1
框架自动处理: 1\' OR \'1\'=\'1（PHP Magic Quotes）
SQL执行: SELECT * FROM table WHERE id = '1\' OR \'1\'=\'1'
结果: 无注入效果
```

**判定逻辑：**
```
IF 输入包含SQL特殊字符
AND 响应正常无SQL语法错误
AND 响应显示查询结果未受影响
AND 推断应用可能使用框架自动安全处理（无法直接确认，通过响应推断）
THEN 误报 = True
```

### 场景10：应用层自定义过滤函数

**特征识别：**
- 应用实现自定义的输入验证/过滤逻辑
- 过滤后参数值发生变化或被拒绝

**与场景9的区别：**
- **场景9**：框架/语言层自动机制，开发者无需显式编码
- **场景10**：应用开发者显式编写的过滤/验证逻辑

**误报判定依据：**
1. 过滤规则识别：

| 过滤类型 | 示例 | 效果 |
|----------|------|------|
| 字符白名单 | 只允许数字/字母 | 过滤所有特殊字符 |
| 关键词黑名单 | 移除`SELECT`等 | 去除SQL关键字 |
| 正则替换 | `s/'//g` → 删除单引号 | 替换单引号为空 |
| 长度限制 | 超过n字符截断 | 破坏注入完整性 |
| 参数验证 | 非法值直接拒绝 | 返回参数错误 |

2. 响应验证：
   - 参数值被过滤后无注入效果
   - 或参数验证失败，返回错误信息

**典型示例：**
```
输入: 1' AND '1'='1
应用过滤后: 1 AND 11（单引号被删除）
或响应: {"error": "参数包含非法字符"}
```

**判定逻辑：**
```
IF 输入包含注入特征
AND 响应显示参数已被处理（过滤、截断等）
或 响应返回参数验证错误
AND 处理后的值无注入能力
THEN 误报 = True
```

## 非误报判定标准

### 真实注入告警特征

若不满足上述任何误报场景，且存在以下明确注入证据，则判定为**真实告警**：

#### 1. UNION查询成功

**特征：**
- 响应体包含查询结果数据
- 响应内容包含原本不存在的数据库信息

**判定：**
```
IF Payload包含UNION SELECT
AND 响应体返回额外数据（如用户表、密码等）
AND 数据结构与预期一致
THEN 真实告警 = True
```

#### 2. 布尔盲注可区分

**特征：**
- 不同条件的Payload产生不同响应
- 响应差异可用于推断数据

**判定：**
```
IF Payload: id=1 AND 1=1 → 响应A
AND Payload: id=1 AND 1=2 → 响应B
AND 响应A ≠ 响应B
THEN 真实告警 = True
```

#### 3. 时间盲注可检测

**特征：**
- 包含延时函数的Payload产生可测量的响应延迟

**判定：**
```
IF Payload包含SLEEP(n)或WAITFOR DELAY
AND 响应时间明显延迟（与n正相关）
AND 延迟时间可控
THEN 真实告警 = True
```

#### 4. 错误注入返回数据库信息

**特征：**
- 响应体包含数据库错误信息
- 错误信息泄露敏感数据（表名、列名、版本等）

**判定：**
```
IF 响应体包含数据库语法错误
AND 错误信息包含结构信息（表名、列名）
AND 错误由注入Payload触发
THEN 真实告警 = True
```

#### 5. 响应包含数据库敏感信息

**特征：**
- 响应体直接包含数据库查询结果
- 数据为用户无权访问的内容

**判定：**
```
IF 响应体包含数据库数据
AND 数据超出正常业务范围
AND 数据由注入Payload获取
THEN 真实告警 = True
```

4. 如无匹配，检查真实注入特征

```
判定: 误报（场景2）
```

### 示例3：误报 - 唯一键冲突

```
请求: POST /api/register
请求体: {"username": "admin"}
响应: {"error": "Duplicate entry 'admin' for key 'PRIMARY'"}
分析: 数据库错误但为唯一键冲突，非SQL语法错误
判定: 误报（场景3）
```

### 示例4：误报 - 动态排序参数

```
请求: GET /api/list?sort=name
检测工具: GET /api/list?sort=name+AND+1=1
响应: {"error": "invalid sort field"}
分析: 合法参数被工具异常Payload替代，服务器拒绝
判定: 误报（场景4）
```

### 示例5：真实告警 - UNION查询成功

```
请求: GET /api/user?id=1' UNION SELECT username,password FROM users--
响应: {"data": [["admin", "password123"], ...]}
分析: UNION查询成功返回用户表数据
判定: 真实告警（UNION查询成功）
```

### 示例6：真实告警 - 布尔盲注

```
请求1: GET /api/user?id=1 AND 1=1 → 响应: {"status": "ok"}
请求2: GET /api/user?id=1 AND 1=2 → 响应: {"status": "error"}
分析: 不同条件产生不同响应，可区分真值
判定: 真实告警（布尔盲注可区分）
```

### 使用说明

1. **触发条件**：当用户需要分析SQL注入类告警时使用此cookbook
2. **分析步骤**：
   - 提取请求体和响应体
   - 识别并解码编码内容
   - 按优先级检查误报场景
   - 如无匹配，检查真实注入特征
   - 输出判定结果和依据

3. **置信度评估**：
   - 高：明确匹配误报场景或真实注入特征
   - 中：部分匹配，需要额外上下文确认
   - 低：无明确特征，需要人工复核

## 注意事项

1. **编码识别优先级**：按URL编码 → Base64 → Hex → Unicode → HTML实体 → ASCII → JSON编码的顺序检测
2. **编码检测流程**：最多解码3层，防止解码循环
3. **上下文分析**：结合URL路径、参数名、业务场景综合判断
4. **响应分析**：重点分析响应体是否包含注入成功的证据
5. **误报场景覆盖**：确保覆盖所有21个误报场景
6. **场景优先级**：高优先级场景优先匹配，低优先级场景需要更多上下文验证
7. **无法判定处理**：对于无法判定的告警，标记为"待人工复核"并记录原因
8. **场景冲突处理**：多场景匹配时选择优先级最高的，或标记为"多场景匹配"
9. **数据库错误判定**：语法错误需结合Payload上下文判断，业务错误需检查Payload是否正常
10. **非SQL语法识别**：特别注意NoSQL查询语法、搜索引擎语法等非SQL场景
