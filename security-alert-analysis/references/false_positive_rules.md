# 微步告警误报判定通用规则

本文档是误报判定的"总则"。具体到攻击类型的判定细节（如 SQL 注入特征、XSS 反射点识别），见 `references/cookbook_*.md`。

## 适用范围

仅对**设备来源字段包含"微步"**的告警做误报判定。绿盟告警不在此流程内——因为本 skill 的目标之一就是用微步的告警补绿盟的覆盖盲区，所以绿盟告警视为基线，微步告警需被审视。

## 判定依据（按优先级）

1. **业务流量识别**：Elasticsearch 监控请求、Redis 配置拉取、健康检查、运维脚本等典型正常流量
2. **编码绕过检测**：URL/Hex/Unicode/Base64/HTML 实体等编码后是否藏匿攻击 payload
3. **攻击特征匹配**：基于 SQL 注入、XSS、RCE、路径遍历等特征库判定
4. **响应体分析**：响应体是否回显敏感信息、错误堆栈、注入结果

## 置信度分级

| 等级 | 触发条件 |
|------|----------|
| 高 | 业务流量特征明确匹配，或攻击特征明确出现 |
| 中 | 存在编码但原始和解码后均无攻击特征；或证据较少 |
| 低 | 业务流量同时夹杂攻击特征；或编码处理结果不确定 |

## 判定结果

| 判定 | 触发条件 |
|------|----------|
| 误报（是） | 匹配业务流量模式 **且** 无攻击特征 |
| 真阳性（否） | 存在明确攻击特征 |
| 需人工审核 | 业务流量同时存在攻击特征；或原始有特征但解码后消失；或证据不足 |

**为什么要"需人工审核"档：** 自动判定不可能 100% 正确，给出明确的不确定档比强行选边更可靠——它把"无法判定"显式化，便于人工聚焦审查。禁止用"是/否"硬填这种边界用例。

## 通用编码绕过识别表

请求体分析时按优先级依次检测：

| 优先级 | 编码类型 | 检测特征 | 解码方法 | 示例 |
|--------|----------|----------|----------|------|
| 1 | URL 编码 | `%[0-9A-Fa-f]{2}` | urldecode | `%27` → `'` |
| 2 | Base64 | `^[A-Za-z0-9+/]+=*$` 且 len%4==0 | base64_decode | `JyBPUg==` → `' OR` |
| 3 | Hex | `^0x[0-9A-Fa-f]+$` 或连续十六进制对 | hex2bin | `0x27` → `'` |
| 4 | Unicode | `\\u[0-9A-Fa-f]{4}` | unicode_decode | `'` → `'` |
| 5 | HTML 实体 | `&#\d+;` 或 `&#x[0-9A-Fa-f]+;` | html_entity_decode | `&#39;` → `'` |
| 6 | ASCII 函数 | `CHAR\(\d+\)` / `ASCII\(\d+\)` | chr | `CHAR(39)` → `'` |
| 7 | JSON 转义 | `\\["'\\\/]` 等 | JSON 解析 | `\"` → `"` |
| 8 | UTF-7 | `+...-` 字符集 | UTF-7 解码 | `+ACs-` → `'` |
| 9 | 多重编码 | 多层编码特征 | 递归解码（≤3 层） | `%2527` → `%27` → `'` |

每层解码后都要重新匹配特征——多重编码是常见绕过方式。最多解码 3 层以防循环。

## 输出字段（每条微步独有告警）

- **置信度**：高 / 中 / 低
- **是否误报**：是（误报） / 否（真阳性） / 需人工审核
- **误报原因说明**：判定依据，引用具体的请求体片段、响应特征或业务流量模式
- **编码绕过检测**：是否检出编码及解码后内容
- **判定证据**：支持本次判定的证据列表
